📐 分布式理论

CAP 定理

分布式系统中，以下三者最多只能同时满足两个：

  • C（Consistency）一致性：所有节点看到相同数据
  • A（Availability）可用性：每个请求都能得到响应
  • P（Partition Tolerance）分区容错：网络分区时系统仍能工作

分布式系统中网络分区不可避免，因此实际选择是 CP 或 AP：

  • CP 系统：ZooKeeper、Etcd、Nacos（持久实例）— 牺牲可用性保一致性
  • AP 系统：Eureka、Nacos（临时实例）、Cassandra — 牺牲一致性保可用性

BASE 理论

对 CAP 中 AP 方案的延伸：

  • Basically Available：基本可用，允许响应时间和功能上的损失
  • Soft State：软状态，允许中间状态存在
  • Eventually Consistent：最终一致性，经过一段时间后数据达到一致

RedLock 算法

为了解决 Redis 主从切换丢锁问题，Redis 作者提出了 RedLock：

  1. 向 N 个独立的 Redis 节点（推荐 5 个）请求加锁
  2. 当超过 N/2+1 个节点加锁成功，且总耗时未超过锁的 TTL → 加锁成功
  3. 锁的有效时间 = TTL - 获取锁的耗时
⚠️ RedLock 争议 Martin Kleppmann 指出 RedLock 在 GC 停顿、时钟跳跃等场景下不安全。实际生产中推荐使用 Redisson（看门狗自动续期）或 ZooKeeper。

Redisson 看门狗机制

  • 默认锁超时时间 30 秒
  • 后台线程每 10 秒（锁时间/3）检查锁是否还被持有
  • 如果业务还没执行完，自动续期到 30 秒
  • 如果客户端宕机，锁到期自动释放（不会死锁）

🔄 分布式事务

方案对比

方案一致性性能复杂度
2PC（XA）强一致低（阻塞）低
TCC（Try-Confirm-Cancel）最终一致高高（需三个接口）
SAGA最终一致高中
本地消息表最终一致高中
Seata AT 模式最终一致较高低

Seata 四种模式

  • AT 模式（推荐）：自动补偿，通过 undo_log 实现回滚，对业务无侵入
  • TCC 模式：手动编写 Try/Confirm/Cancel，灵活但侵入性大
  • SAGA 模式：长事务方案，每个参与者有正向和反向操作
  • XA 模式：基于数据库 XA 协议，强一致性但性能差

本地消息表方案

  1. 业务操作和写消息表在同一个本地事务中
  2. 定时任务扫描消息表，发送到 MQ
  3. 下游消费 MQ 并处理，处理成功后更新消息状态
  4. 失败重试，保证最终一致性

消息丢失问题

三个环节都可能丢消息，需要分别处理：

  1. 生产者 → MQ：开启确认机制（RabbitMQ confirm / Kafka acks=all）
  2. MQ 自身：持久化消息 + 刷盘策略（同步刷盘）+ 集群多副本
  3. MQ → 消费者：手动 ACK，消费成功后再确认

消息重复消费（幂等性）

MQ 不能保证 exactly-once，只能做到 at-least-once，因此消费端需要做幂等：

  • 数据库唯一索引：同一消息不会插入两次
  • Redis SETNX：记录已处理的消息 ID
  • 状态机：业务状态只能单向流转（如 待支付 → 已支付，不会重复扣款）
  • 乐观锁：UPDATE SET status = 'paid' WHERE id = ? AND status = 'unpaid'

消息顺序性

全局有序性能太差，一般保证分区有序：

  • 同一业务 key（如订单 ID）的消息发到同一个分区/Queue
  • 消费者单线程消费或加锁处理
💡 延迟队列实现 ① RocketMQ 原生支持延迟消息；② RabbitMQ 通过 TTL + 死信队列实现；③ Redis 用 ZSet + 时间戳轮询实现；④ 时间轮（HashedWheelTimer）适用于大量定时任务。

🎯 经典系统设计

秒杀系统设计

核心挑战：瞬时高并发、超卖问题、恶意刷单。

整体架构

  1. 前端：按钮置灰、验证码、URL 动态化（防止脚本刷）
  2. Nginx：限流（漏桶/令牌桶）、负载均衡
  3. 网关层：用户限流（同一用户 N 次/秒）、黑名单
  4. Service 层：
    • Redis 预减库存（DECR 原子操作，库存 ≤ 0 直接返回）
    • 内存标记（Map 标记售罄，减少 Redis 访问）
    • 异步下单：发送 MQ 消息
  5. 消费端：MQ 消费者执行真正的数据库扣减和订单创建
  6. 数据库：乐观锁防超卖 UPDATE SET stock = stock - 1 WHERE id = ? AND stock > 0
// 秒杀核心逻辑
public Result seckill(Long userId, Long itemId) {
    // 1. 内存标记，过滤已售罄请求
    if (soldOutMap.get(itemId)) return Result.fail("已售罄");

    // 2. Redis 预减库存
    long stock = redisTemplate.opsForValue().decrement("stock:" + itemId);
    if (stock < 0) {
        soldOutMap.put(itemId, true);
        redisTemplate.opsForValue().increment("stock:" + itemId);
        return Result.fail("已售罄");
    }

    // 3. 判断是否重复抢购
    if (redisTemplate.opsForSet().isMember("bought:" + itemId, userId)) {
        return Result.fail("请勿重复购买");
    }

    // 4. 发送 MQ 异步下单
    mqProducer.sendSeckillMessage(userId, itemId);
    return Result.ok("排队中，请稍候");
}

短链接系统设计

  • 生成方案：分布式 ID → Base62 编码 → 6-8 位短码
  • 存储：MySQL 存长短映射 + Redis 缓存热点短链
  • 跳转：302 重定向（可统计点击量）或 301（搜索引擎友好）
  • 防重复：对原始 URL 做 MD5 取前 N 位查询是否已存在

接口幂等设计

保证同一个请求执行多次与执行一次效果相同：

  1. Token 机制：请求前获取 Token，提交时携带 Token 并删除（一次性）
  2. 唯一请求 ID：客户端生成唯一 ID，服务端用 Redis SETNX 去重
  3. 数据库唯一约束：利用唯一索引防止重复插入
  4. 乐观锁：版本号机制 WHERE version = ?

🧩 设计模式

面试常考的设计模式

单例模式

// 推荐：枚举单例（天然防反射和序列化攻击）
public enum Singleton {
    INSTANCE;

    public void doSomething() { /* ... */ }
}

// 双重检查锁（DCL）
public class Singleton {
    private static volatile Singleton instance;
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) instance = new Singleton();
            }
        }
        return instance;
    }
}

// 消除 if-else，Spring 中常用

public class PayStrategyFactory {

模板方法模式

// 定义算法骨架，子类实现具体步骤
public abstract class AbstractExportService {
    public final void export() {
        validate();       // 通用校验
        List<Data> data = queryData();  // 子类实现
        process(data);    // 子类实现
        generateFile();   // 通用生成
    }
    protected abstract List<Data> queryData();
    protected abstract void process(List<Data> data);
}

观察者模式

Spring 事件机制就是典型的观察者模式：

// 定义事件
public class OrderCreatedEvent extends ApplicationEvent {
    private final Order order;
    public OrderCreatedEvent(Object source, Order order) {
        super(source);
        this.order = order;
    }
}

// 监听事件
@Component
public class OrderEventListener {
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        // 发送通知、更新统计等
    }
}

SOLID 原则

  • S（单一职责）：一个类只负责一件事
  • O（开闭原则）：对扩展开放，对修改关闭
  • L（里氏替换）：子类可以替换父类
  • I（接口隔离）：接口要小而专
  • D（依赖倒置）：依赖抽象而非具体实现

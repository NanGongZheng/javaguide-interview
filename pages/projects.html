🛒 电商秒杀系统

项目简介

面向大促场景的高并发秒杀系统，支持商品秒杀、限时抢购、优惠券发放等场景。系统需在极短时间内承受数十万级并发请求，同时保证库存准确性和数据一致性。

技术栈

层级技术选型说明
前端Vue 3 + Axios按钮防重、倒计时、动态 URL
网关Spring Cloud Gateway限流、鉴权、请求路由
服务层Spring Boot 3 + MyBatis-Plus业务逻辑、库存扣减
缓存Redis + Lua 脚本库存预热、原子扣减、分布式锁
消息队列RocketMQ异步下单、削峰填谷
数据库MySQL 8 + 分库分表（ShardingSphere）订单分表、乐观锁防超卖
限流Sentinel接口限流、熔断降级
部署Docker + K8s弹性扩缩容

项目亮点

  • 多级缓存架构：本地缓存（Caffeine）+ Redis 缓存 + 数据库，QPS 从 500 提升至 8000+
  • Redis + Lua 原子扣减：库存预热到 Redis，通过 Lua 脚本保证"判断 + 扣减"的原子性，避免超卖
  • 异步下单：秒杀请求只做 Redis 扣减，通过 RocketMQ 异步创建订单，削峰填谷
  • 多层限流防护：前端按钮置灰 → Gateway 令牌桶限流 → Sentinel 接口限流 → Redis 用户级限流
  • 库存售罄标记：Redis 内存标记售罄后，后续请求直接返回，不再访问 Redis

难题与解决方案

难题一：高并发下库存超卖问题 ▼

问题描述：使用 GET + SET 判断库存并扣减时，并发请求可能导致超卖。

解决方案：

-- Redis Lua 脚本：原子判断 + 扣减
local stock = redis.call('get', KEYS[1])
if not stock or tonumber(stock) <= 0 then
    return -1  -- 库存不足
end
redis.call('decr', KEYS[1])
return 1  -- 扣减成功

配合数据库乐观锁兜底：UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0

难题二：秒杀结束后大量无效请求涌入 ▼

问题描述：商品售罄后仍有大量请求到达 Redis，浪费资源。

解决方案：引入内存标记机制——用一个 ConcurrentHashMap<Long, Boolean> 标记售罄商品，请求先查内存标记，命中则直接返回"已售罄"，不再访问 Redis。售罄时设置标记并设置 Redis Key 过期时间。

难题三：消息丢失导致订单未创建 ▼

问题描述：Redis 扣减成功后，MQ 消息丢失导致订单未创建，但库存已扣。

解决方案：采用 RocketMQ 事务消息机制：

  1. 发送 Half 消息（预消息）
  2. 执行本地事务（创建订单记录到"事务日志表"）
  3. 本地事务成功 → Commit，MQ 投递消息给消费者
  4. 本地事务失败 → Rollback，库存回补
  5. MQ 定时回查未确认的 Half 消息，检查事务日志表决定 Commit 或 Rollback

  • R（结果）：大促期间 QPS 峰值 8000+，零超卖，P99 延迟 < 200ms，系统零故障

💬 即时通讯系统

项目简介

企业级 IM 即时通讯系统，支持单聊、群聊、已读回执、消息漫游、文件传输等功能。类似企业微信/飞书的核心通讯能力。

技术栈

层级技术选型说明
通信协议WebSocket（Netty 实现）全双工实时通信
服务层Spring Boot + MyBatis-Plus用户管理、群组管理
消息存储MySQL + MongoDB关系数据 + 聊天消息（文档型）
缓存Redis在线状态、会话列表、未读数
文件存储MinIO / 阿里云 OSS图片、文件、语音消息
消息推送RocketMQ离线消息推送、消息同步
搜索Elasticsearch聊天记录全文检索

项目亮点

  • Netty 长连接管理：基于 Netty 实现 WebSocket 服务端，单机支撑 10 万+ 长连接，心跳检测 + 自动重连
  • 消息可靠性三件套：ACK 确认机制 + 消息重试 + 消息去重（幂等 ID），保证消息不丢不重
  • 消息有序性：使用 Snowflake 算法生成全局有序消息 ID，配合 SequenceId 保证会话内消息有序
  • 多端同步：用户在 PC/手机/iPad 同时在线，消息通过"同步序列号"机制多端同步
  • 已读回执：Redis Bitmap 记录群消息已读状态，O(1) 查询某条消息的已读人数

难题与解决方案

难题一：百万长连接下的内存与性能瓶颈 ▼

问题描述：单机 Netty 连接数增长后，内存占用飙升，GC 频繁。

解决方案：

  • 内存池化：使用 Netty 的 PooledByteBufAllocator，减少 GC 压力
  • 零拷贝：文件传输使用 FileRegion，避免用户态拷贝
  • 连接分桶：按用户 ID 哈希分配到不同的 EventLoopGroup，避免单线程过载
  • 心跳优化：空闲连接检测（IdleStateHandler），超时自动断开，释放资源
难题二：离线用户消息如何不丢失 ▼

问题描述：用户离线期间收到的消息，上线后需要完整获取。

解决方案：

  1. 为每个用户维护一个 收件箱（Inbox），记录最新已同步的消息序列号 syncSeq
  2. 消息发送时同时写入"消息表"和"收件箱表"
  3. 用户上线时，用本地最大 syncSeq 向服务端拉取增量消息
  4. 离线消息暂存 MongoDB，按会话分片，支持海量存储

面试 STAR 描述模板

💡 STAR 法则
  • S（背景）：公司需要自建 IM 系统替代第三方 SDK，降低长期成本并满足数据合规要求
  • T（任务）：负责 IM 核心通信层设计，支撑 50 万在线用户、消息秒级送达
  • A（行动）：基于 Netty 构建 WebSocket 长连接服务，设计 ACK + 重试 + 去重三件套保证消息可靠性，Redis Bitmap 实现已读回执
  • R（结果）：单机 10 万长连接，消息送达率 99.99%，已读回执查询 P99 < 10ms

🍱 外卖点餐平台

项目简介

类似美团外卖的在线点餐平台，涵盖用户端（浏览商家、下单、支付）、商家端（接单、出餐）、骑手端（接单、配送）三端。核心挑战是订单状态机和高并发下的数据一致性。

技术栈

层级技术选型说明
前端微信小程序 + Vue 3 管理端用户端小程序 + 商家/骑手管理后台
网关Spring Cloud Gateway路由、鉴权、限流
服务层Spring Boot 3 + MyBatis-Plus用户服务、商家服务、订单服务、配送服务
注册/配置Nacos服务注册发现 + 动态配置
数据库MySQL（分库分表）订单表按 user_id 分片
缓存Redis Cluster商家列表缓存、购物车、地理位置（GEO）
消息队列RocketMQ订单超时取消、商家通知
地图服务高德地图 API距离计算、配送路线规划
支付微信支付 + 支付宝统一支付网关封装

项目亮点

  • 订单状态机：使用 Spring StateMachine 实现订单状态流转（待支付→已支付→商家接单→出餐中→配送中→已完成），每一步都有严格的状态校验和事件驱动
  • Redis GEO 附近商家：利用 Redis GEO 命令实现"附近 3 公里商家"查询，GEOSEARCH 性能优于数据库经纬度范围查询
  • 延迟队列自动取消：用户下单未支付，30 分钟后通过 RocketMQ 延迟消息自动取消订单并释放库存
  • 购物车设计：Redis Hash 存储购物车（key 为 userId，field 为 skuId），支持商品规格、数量修改
  • 统一支付网关：封装微信/支付宝支付接口，统一回调处理，支付结果通过 MQ 幂等消费

难题与解决方案

难题一：订单状态流转的并发安全问题 ▼

问题描述：用户取消和商家接单可能同时发生，导致状态混乱。

解决方案：

  • 乐观锁：更新订单时带状态条件 UPDATE orders SET status = ? WHERE id = ? AND status = ?，影响行数为 0 则说明状态已变更
  • 状态机校验：每次状态变更前校验"当前状态 → 目标状态"是否合法，非法转换直接拒绝
  • 分布式锁兜底：关键操作（如支付回调）加 Redis 分布式锁，防止重复处理
难题二：高峰时段商家接单延迟 ▼

问题描述：午高峰大量订单涌入，商家端 WebSocket 推送延迟严重。

解决方案：

  • 消息分级：新订单用"高优先级队列"单独处理，与普通消息隔离
  • 本地缓存 + 增量推送：商家端维护本地订单列表，WebSocket 只推送增量变更（订单 ID + 状态）
  • 降级方案：WebSocket 不可用时自动降级为轮询（每 5 秒 HTTP 拉取）

面试 STAR 描述模板

💡 STAR 法则
  • S（背景）：公司拓展外卖业务，需要从零搭建点餐平台
  • T（任务）：负责订单核心模块，覆盖下单、支付、状态流转、超时取消全流程
  • A（行动）：设计状态机驱动的订单系统，Redis GEO 实现附近商家，RocketMQ 延迟消息实现超时取消
  • R（结果）：日均订单 5 万+，午高峰 QPS 2000+，订单状态零异常，用户投诉率降低 60%

🔗 短链接服务

项目简介

高性能短链接生成与跳转服务，类似 bit.ly / t.cn。支持短链生成、302 跳转、访问统计、自定义短码、过期管理等功能。核心挑战是高并发跳转下的低延迟和海量数据存储。

技术栈

层级技术选型说明
服务层Spring Boot 3 + MyBatis-Plus短链生成、跳转、统计
缓存Redis Cluster + 本地缓存（Caffeine）热点短链两级缓存
数据库MySQL + ShardingSphere短链映射表按短码哈希分片
ID 生成雪花算法（自研）全局唯一、趋势递增的 ID
监控Prometheus + Grafana跳转 QPS、缓存命中率监控
消息队列Kafka访问日志异步采集

项目亮点

  • Base62 编码方案：自增 ID + Base62 编码生成 6~8 位短码，6 位可表示 568 亿种组合，碰撞概率极低
  • 两级缓存架构：Caffeine 本地缓存（容量 10 万，TTL 5 分钟）+ Redis 缓存，缓存命中率 99.5%+
  • 布隆过滤器防穿透：使用 Redis Bloom 过滤不存在的短码，避免缓存穿透打到数据库
  • 异步统计：访问日志写入 Kafka，Flink 实时聚合统计（PV/UV/地域分布），不影响跳转性能
  • 短码回收：过期短码不立即删除，进入"冷却池"一段时间后回收复用，避免短码资源浪费

难题与解决方案

难题一：缓存穿透导致数据库压力飙升 ▼

问题描述：恶意用户大量请求不存在的短码，请求穿透到数据库。

解决方案：

  • 布隆过滤器：启动时加载所有有效短码到 Redis Bloom，请求先过布隆过滤器，不存在直接返回 404
  • 空值缓存：查询不存在的短码也缓存空值（TTL 60 秒），防止短时间内重复穿透
  • 限流：同一 IP 每秒最多 100 次请求，超限返回 429
难题二：分布式环境下短码生成冲突 ▼

问题描述：多实例同时生成短码，ID 冲突导致短码重复。

解决方案：

  • 雪花算法：每个实例分配唯一 WorkerId，生成的 ID 全局唯一且趋势递增
  • 号段模式兜底：引入 Leaf 号段模式，从数据库预取号段到本地，减少 DB 依赖
  • 冲突检测：生成短码后插入数据库（唯一索引），冲突则重新生成（概率极低）

面试 STAR 描述模板

💡 STAR 法则
  • S（背景）：营销部门大量使用短链接做推广投放，原有第三方服务费用高且无法定制
  • T（任务）：自研短链接服务，日均跳转量 500 万+，P99 延迟 < 5ms
  • A（行动）：设计 Base62 编码 + 雪花算法 ID 生成，两级缓存 + 布隆过滤器防穿透，Kafka 异步统计
  • R（结果）：跳转 QPS 1 万+，P99 延迟 3ms，缓存命中率 99.5%，年节省第三方服务费 10 万+

🔐 企业级权限管理系统

项目简介

基于 RBAC（Role-Based Access Control）模型的企业级权限管理系统，支持用户管理、角色管理、菜单权限、数据权限、操作日志审计等功能。作为公司内部多个业务系统的统一权限底座。

技术栈

层级技术选型说明
前端Vue 3 + Element Plus动态菜单、权限按钮控制
安全框架Spring Security + JWT认证授权、Token 无状态鉴权
服务层Spring Boot 3 + MyBatis-Plus用户/角色/菜单/权限管理
数据库MySQL 8用户表、角色表、菜单表、关联表
缓存Redis用户权限缓存、Token 黑名单
日志ELK（Elasticsearch + Logstash + Kibana）操作日志采集与检索
接口文档Knife4j（Swagger 增强）在线 API 文档

项目亮点

  • RBAC 权限模型：用户 → 角色 → 权限三层模型，支持菜单权限（页面可见性）和按钮权限（操作控制）
  • 数据权限：基于 MyBatis 拦截器实现数据权限过滤（本人数据 / 本部门数据 / 全部数据），无需业务代码侵入
  • 动态菜单：前端路由根据用户权限动态生成，后端返回菜单树，前端递归渲染侧边栏
  • JWT 无状态鉴权：Token 存储用户 ID + 角色列表，网关统一校验，无需 Session 共享
  • 操作审计日志：自定义注解 @AuditLog + AOP 切面，自动记录谁在什么时间做了什么操作

难题与解决方案

难题一：数据权限如何无侵入地实现 ▼

问题描述：不同角色看到的数据范围不同（如销售只能看自己的客户，主管看部门全部），但不想在每个 SQL 里硬编码权限条件。

解决方案：

  1. 自定义 @DataScope 注解，标注在 Mapper 方法上
  2. MyBatis 拦截器（Interceptor）拦截查询 SQL
  3. 根据当前用户的角色和数据权限级别，自动拼接 WHERE dept_id = ? 或 WHERE create_by = ?
  4. 业务代码完全无感知，只需加一个注解即可
难题二：用户权限变更后 Token 如何即时生效 ▼

问题描述：JWT 是无状态的，用户角色被修改后，已签发的 Token 仍然有效，权限不会即时更新。

解决方案：

  • Token 版本号：用户表维护一个 token_version 字段，JWT 中携带版本号，校验时比对版本号，不匹配则拒绝
  • Redis 黑名单：权限变更时将旧 Token 加入 Redis 黑名单（TTL 等于 Token 有效期），网关校验时先查黑名单
  • 权限缓存刷新：权限变更后主动清除该用户的 Redis 权限缓存，下次请求重新加载

面试 STAR 描述模板

💡 STAR 法则
  • S（背景）：公司有 10+ 业务系统，各自维护权限体系，管理混乱且存在安全隐患
  • T（任务）：设计统一权限管理平台，支持菜单权限 + 数据权限 + 操作审计

  • R（结果）：接入 8 个业务系统，统一权限管理，权限变更秒级生效，操作日志覆盖率 100%

💡 项目面试通用技巧

如何讲好项目故事

  • 先说业务背景：面试官不了解你的项目，先用 1-2 句话描述业务场景和核心目标
  • 突出你的贡献：用"我负责...""我设计...""我优化..."，而不是"我们团队..."
  • 用数据说话：QPS 从 X 提升到 Y、延迟从 A 降到 B、错误率从 C% 降到 D%
  • 讲清权衡取舍：为什么选 A 方案而不是 B 方案？有什么 trade-off？
  • 准备"踩坑"故事：遇到什么难题？怎么排查的？最终怎么解决的？

☕ Java 基础

HashMap 的底层实现原理是什么？ 中频 ▼

JDK 8 中 HashMap 采用 数组 + 链表 + 红黑树 结构：

  • 默认容量 16，负载因子 0.75，当元素数量超过 12（16×0.75）时扩容为原来的 2 倍
  • 通过 (n-1) & hash 定位桶位置，hash 值的高 16 位异或低 16 位减少碰撞
  • 链表长度 ≥ 8 且数组长度 ≥ 64 时转红黑树；红黑树节点 ≤ 6 时退化为链表
  • JDK 8 用尾插法替代 JDK 7 的头插法，解决了多线程扩容时的环形链表问题
  • HashMap 非线程安全，并发场景使用 ConcurrentHashMap
ConcurrentHashMap 如何保证线程安全？ 中频 ▼

JDK 8 中 ConcurrentHashMap 采用 CAS + synchronized：

  • 初始化：CAS 操作设置 sizeCtl
  • put：桶为空时 CAS 插入；桶不为空时 synchronized 锁住头节点
  • 扩容：多线程协助迁移（transfer），每个线程负责一段
  • size()：baseCount + CounterCell[] 分散计数，类似 LongAdder

对比 JDK 7 的 Segment 分段锁，锁粒度从段级别降到桶级别，并发度更高。

synchronized 和 ReentrantLock 有什么区别？ 中频 ▼
  • 实现：synchronized 是 JVM 关键字（monitorenter/monitorexit），ReentrantLock 是 API 层面（基于 AQS）
  • 锁释放：synchronized 自动释放，ReentrantLock 需手动 unlock（finally 中）
  • 可中断：synchronized 不可中断，ReentrantLock 可通过 lockInterruptibly() 中断
  • 公平锁：synchronized 只有非公平，ReentrantLock 支持公平/非公平
  • 条件变量：synchronized 用 wait/notify（单条件），ReentrantLock 用 Condition（多条件）
  • 锁升级：synchronized 有偏向锁→轻量级锁→重量级锁的升级过程
volatile 关键字的作用和原理？ 高频 ▼

volatile 保证可见性和有序性，不保证原子性：

  • 可见性：修改后立即刷新到主内存，其他线程读取时从主内存加载（通过内存屏障实现）
  • 有序性：禁止指令重排序（写前加 StoreStore 屏障，读后加 LoadLoad 屏障）
  • 不保证原子性：count++ 是 read-modify-write 三步操作，多线程下仍需加锁或用 AtomicInteger

典型应用：DCL 单例中防止指令重排、状态标志位。

线程池的核心参数有哪些？任务提交流程是什么？ 高频 ▼

七个核心参数：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（非核心线程存活时间）、unit（时间单位）、workQueue（任务队列）、threadFactory（线程工厂）、handler（拒绝策略）。

任务提交流程：

  1. 线程数 < corePoolSize → 创建核心线程执行
  2. 核心线程已满 → 任务入 workQueue
  3. 队列已满 → 创建非核心线程（不超过 maximumPoolSize）
  4. 线程和队列都满 → 执行拒绝策略

四种拒绝策略：AbortPolicy（抛异常）、CallerRunsPolicy（调用者执行）、DiscardPolicy（丢弃）、DiscardOldestPolicy（丢弃最老）。

JVM 垃圾回收算法有哪些？G1 收集器的特点？ 中频 ▼

四种基础算法：

  • 标记-清除：先标记后清除，产生内存碎片
  • 标记-整理：标记后移动存活对象，无碎片但开销大
  • 复制算法：分两块，存活对象复制到另一块，浪费空间
  • 分代收集：新生代（复制）+ 老年代（标记整理），JVM 实际策略

G1 收集器特点：

  • 将堆划分为多个等大的 Region，每个 Region 可以是 Eden/Survivor/Old/Humongous
  • 可预测停顿时间（-XX:MaxGCPauseMillis），优先回收价值最大的 Region
  • JDK 9+ 默认收集器，适合大堆（6GB+）

🌱 Spring

Spring Boot 自动配置原理是什么？ 高频 ▼
  1. @SpringBootApplication 组合了 @EnableAutoConfiguration
  2. @EnableAutoConfiguration 通过 @Import(AutoConfigurationImportSelector) 导入配置
  3. AutoConfigurationImportSelector 读取 META-INF/spring.factories 中的自动配置类
  4. 每个配置类通过 @Conditional 系列注解判断是否生效（@ConditionalOnClass、@ConditionalOnMissingBean 等）
  5. 生效的配置类向容器注册 Bean

Spring Boot 3.x 改用 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件。

Spring Bean 的生命周期是怎样的？ 高频 ▼
  1. 实例化（构造器 / 工厂方法）
  2. 属性赋值（@Autowired、@Value）
  3. Aware 回调（BeanNameAware → BeanFactoryAware → ApplicationContextAware）
  4. BeanPostProcessor#postProcessBeforeInitialization
  5. 初始化（@PostConstruct → InitializingBean → init-method）
  6. BeanPostProcessor#postProcessAfterInitialization（AOP 代理在此创建）
  7. 使用 Bean
  8. 销毁（@PreDestroy → DisposableBean → destroy-method）
Spring 事务失效的常见场景有哪些？ 高频 ▼
  1. 方法非 public：Spring AOP 只能代理 public 方法
  2. 自调用：同类中 A 调 B，B 的 @Transactional 不生效（未走代理）
  3. 异常被 catch：吞掉异常后 Spring 感知不到
  4. 抛出非 RuntimeException：默认只回滚 RuntimeException 和 Error
  5. 传播行为设置错误：如 NOT_SUPPORTED
  6. rollbackFor 未指定：需要回滚 checked 异常时需显式声明
  7. 数据库引擎不支持：MyISAM 不支持事务
  8. 多线程调用：新线程不在同一个事务上下文中
Spring 循环依赖是如何解决的？ 中频 ▼

Spring 通过三级缓存解决 singleton 作用域的 setter/field 注入循环依赖：

  • 一级缓存 singletonObjects：完全初始化好的 Bean
  • 二级缓存 earlySingletonObjects：提前曝光的 Bean（可能是代理对象）
  • 三级缓存 singletonFactories：ObjectFactory，用于创建早期引用

不能解决的场景：构造器注入的循环依赖（对象还没创建完）。Spring Boot 2.6+ 默认禁止循环依赖。

AOP 的实现原理是什么？JDK 动态代理和 CGLIB 的区别？ 中频 ▼
  • JDK 动态代理：基于接口，通过 Proxy.newProxyInstance() 生成代理类，只能代理实现了接口的类
  • CGLIB 代理：基于继承，通过字节码技术生成目标类的子类，不需要接口，但不能代理 final 类和方法
  • Spring 选择策略：有接口默认用 JDK 代理，无接口用 CGLIB。Spring Boot 2.x 默认全部使用 CGLIB
  • 性能对比：JDK 代理创建快、调用慢；CGLIB 创建慢、调用快。现代 JDK 版本中差距很小

🗄️ 数据库

MySQL 索引为什么使用 B+ 树而不是 B 树或红黑树？ 高频 ▼
  • vs B 树：B+ 树非叶子节点不存数据，单个节点能存更多 key，树更矮，磁盘 IO 更少；叶子节点用链表相连，范围查询高效
  • vs 红黑树：红黑树是二叉树，树高远大于 B+ 树（千万数据可能 20+ 层），磁盘 IO 太多
  • vs Hash 索引：Hash 不支持范围查询和排序，只适合等值查询

InnoDB 中千万级数据的 B+ 树通常只有 3-4 层，每次查询只需 3-4 次磁盘 IO。

MVCC 的原理是什么？RC 和 RR 隔离级别有什么区别？ 高频 ▼

MVCC（多版本并发控制）通过 undo log 版本链 + Read View 实现无锁读：

  • 每行有隐藏字段 DB_TRX_ID（修改事务ID）和 DB_ROLL_PTR（回滚指针）
  • undo log 通过 ROLL_PTR 串成版本链
  • 事务开始时创建 Read View，包含活跃事务ID列表，用于判断版本可见性

RC vs RR：

  • READ COMMITTED：每次 SELECT 创建新 Read View
  • REPEATABLE READ：只在第一次 SELECT 创建 Read View，之后复用
如何优化慢 SQL？ 高频 ▼
  1. 开启慢查询日志定位问题 SQL（slow_query_log）
  2. EXPLAIN 分析执行计划，关注 type、key、rows、Extra
  3. 优化方向：
    • 添加合适索引，优先覆盖索引避免回表
    • 避免 SELECT *，只查需要的字段
    • 避免索引失效写法（函数、隐式转换、LIKE %前缀）
    • 大分页优化：游标分页 WHERE id > lastId LIMIT 20
    • JOIN 优化：小表驱动大表，被驱动表关联字段加索引
    • 避免在 WHERE 中对字段做运算
Redis 缓存穿透、击穿、雪崩分别是什么？如何解决？ 高频 ▼
  • 穿透：查询不存在的数据。解决：布隆过滤器 + 缓存空值
  • 击穿：热点 key 过期瞬间大量请求。解决：互斥锁（setnx）+ 逻辑过期
  • 雪崩：大量 key 同时过期或 Redis 宕机。解决：过期时间加随机值 + 集群高可用 + 限流降级
Redis 和 MySQL 数据一致性如何保证？ 中频 ▼

推荐 Cache Aside Pattern：读先读缓存，miss 则读 DB 并写缓存；写先更新 DB，再删缓存。

延迟双删方案：先删缓存 → 更新 DB → 延迟 N 毫秒 → 再删缓存。应对并发读写时旧值被写入缓存的问题。

更强一致性方案：监听 binlog（Canal）→ 更新/删除缓存，实现最终一致性。

🏗️ 分布式

分布式锁有哪些实现方案？各自的优缺点？ 高频 ▼
  • Redis（SET NX EX + Lua 解锁）：性能高，推荐用 Redisson（看门狗自动续期）。缺点：主从切换可能丢锁
  • ZooKeeper（临时顺序节点）：强一致，自动释放。缺点：性能较低，依赖 ZK 集群
  • 数据库（唯一索引 / FOR UPDATE）：简单无额外依赖。缺点：性能差，死锁风险

生产环境推荐 Redisson 或 ZooKeeper。

如何保证消息不丢失？如何保证消息不重复消费？ 高频 ▼

消息不丢失（三个环节）：

  • 生产者 → MQ：开启确认机制（confirm/acks=all）
  • MQ 自身：持久化 + 同步刷盘 + 多副本
  • MQ → 消费者：手动 ACK，消费成功后确认

幂等性（防重复消费）：

  • 数据库唯一索引
  • Redis SETNX 记录已处理消息 ID
  • 状态机（业务状态只能单向流转）
  • 乐观锁 WHERE version = ?
CAP 理论和 BASE 理论分别是什么？ 中频 ▼

CAP：分布式系统最多同时满足一致性（C）、可用性（A）、分区容错性（P）中的两个。分布式系统 P 不可避免，实际选择是 CP 或 AP。

BASE：基本可用（BA）+ 软状态（S）+ 最终一致性（E），是 CAP 中 AP 方案的延伸，通过牺牲强一致性换取可用性。

秒杀系统如何设计？ 中频 ▼

分层防护：

  1. 前端：按钮置灰、验证码、URL 动态化
  2. Nginx：限流、负载均衡
  3. 网关：用户限流、黑名单
  4. Service：Redis 预减库存（DECR）→ 内存标记售罄 → 发送 MQ 异步下单
  5. 数据库：乐观锁防超卖 WHERE stock > 0

核心思想：尽量把请求拦截在上游，真正到达数据库的请求极少。

💡 场景与经验

你做过的最有挑战的项目是什么？遇到过什么技术难点？ 高频 ▼

回答这类问题建议使用 STAR 法则：

  • S（Situation）：项目的背景和目标
  • T（Task）：你的职责和具体任务
  • A（Action）：你采取的技术方案和行动
  • R（Result）：最终效果和数据指标

示例："负责 XX 系统的高并发改造（S），需要支撑日均百万级订单（T），通过引入 Redis 缓存 + MQ 异步 + 数据库分库分表（A），QPS 从 500 提升到 5000，P99 延迟从 2s 降到 200ms（R）。"

线上 CPU 飙高怎么排查？ 中频 ▼ 
# 1. 找到 CPU 高的 Java 进程
top -c
# 2. 找到该进程中 CPU 高的线程
top -Hp <pid>
# 3. 将线程 ID 转为 16 进制
printf "%x\n" <tid>
# 4. 用 jstack 导出线程栈，搜索该线程
jstack <pid> | grep <tid_hex> -A 30

常见原因：死循环、频繁 Full GC、正则回溯、序列化/反序列化性能问题。

线上 OOM 怎么排查？ 中频 ▼
  1. 启动时加 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof
  2. 用 MAT（Memory Analyzer Tool）或 VisualVM 分析 dump 文件
  3. 查看 Dominator Tree 找到占用内存最大的对象
  4. 分析 GC Roots 引用链，定位泄漏代码

常见原因：大集合未清理、ThreadLocal 未 remove、数据库连接未关闭、缓存无限增长。

如何设计一个接口限流方案？ 中频 ▼

常见限流算法：

  • 固定窗口计数器：简单但有边界突发问题
  • 滑动窗口计数器：更平滑，Sentinel 使用此算法
  • 漏桶算法：恒定速率处理，削峰填谷
  • 令牌桶算法：允许一定程度的突发流量（Guava RateLimiter）

实现方案：Guava RateLimiter（单机）、Redis + Lua（分布式）、Sentinel（生产推荐，支持集群限流）。

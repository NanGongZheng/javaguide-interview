🔄 IoC 与依赖注入

IoC 容器原理

IoC（控制反转）是 Spring 的核心思想，将对象的创建和依赖管理交给 Spring 容器。核心接口：

  • BeanFactory：最基础的容器接口，懒加载 Bean
  • ApplicationContext：BeanFactory 的子接口，功能更丰富（国际化、事件发布、AOP 等）

Bean 生命周期（面试高频）

  1. 实例化：通过构造器或工厂方法创建 Bean 实例
  2. 属性赋值：注入依赖（@Autowired、@Value 等）
  3. Aware 回调：BeanNameAware → BeanFactoryAware → ApplicationContextAware
  4. BeanPostProcessor#postProcessBeforeInitialization
  5. 初始化：@PostConstruct → InitializingBean#afterPropertiesSet → 自定义 init-method
  6. BeanPostProcessor#postProcessAfterInitialization（AOP 代理在此阶段创建）
  7. 使用 Bean
  8. 销毁：@PreDestroy → DisposableBean#destroy → 自定义 destroy-method

Bean 的作用域

作用域说明
singleton（默认）容器中只有一个实例
prototype每次获取都创建新实例
request每个 HTTP 请求一个实例（Web）
session每个 HTTP Session 一个实例（Web）

循环依赖解决

Spring 通过 三级缓存 解决 singleton 作用域的 setter/field 注入循环依赖：

  • 一级缓存 singletonObjects：完全初始化好的 Bean
  • 二级缓存 earlySingletonObjects：提前曝光的 Bean（可能是代理对象）
  • 三级缓存 singletonFactories：Bean 工厂（用于创建早期引用）
⚠️ 不能解决的循环依赖 构造器注入的循环依赖无法解决（因为对象还没创建完）。Spring Boot 2.6+ 默认禁止循环依赖。

✂️ AOP 面向切面编程

AOP 核心概念

  • 切面（Aspect）：横切关注点的模块化（如日志、事务）
  • 连接点（JoinPoint）：程序执行的某个位置（Spring AOP 只支持方法级别的连接点）
  • 切入点（Pointcut）：匹配连接点的表达式
  • 通知（Advice）：在切入点执行的动作（Before、After、Around、AfterReturning、AfterThrowing）

AOP 实现原理

  • JDK 动态代理：基于接口，通过 Proxy.newProxyInstance() 生成代理类
  • CGLIB 代理：基于继承，通过字节码生成目标类的子类
  • Spring 默认：有接口用 JDK 代理，无接口用 CGLIB（Spring Boot 2.x 默认全部使用 CGLIB）
// AOP 实战：统一日志切面
@Aspect
@Component
public class LogAspect {

    @Around("execution(* com.example.service.*.*(..))")
    public Object log(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed();
            long cost = System.currentTimeMillis() - start;
            log.info("{}.{} cost {}ms",
                pjp.getTarget().getClass().getSimpleName(),
                pjp.getSignature().getName(), cost);
            return result;
        } catch (Throwable e) {
            log.error("Method error: {}", e.getMessage());
            throw e;
        }
    }
}

🚀 Spring Boot

自动配置原理（核心面试题）

Spring Boot 自动配置的入口是 @SpringBootApplication 注解，它组合了：

  • @SpringBootConfiguration：本质是 @Configuration
  • @EnableAutoConfiguration：开启自动配置
  • @ComponentScan：包扫描

自动配置流程

  1. @EnableAutoConfiguration 通过 @Import(AutoConfigurationImportSelector.class) 导入配置
  2. AutoConfigurationImportSelector 读取 META-INF/spring.factories（或 Spring Boot 3.x 的 AutoConfiguration.imports）中的自动配置类全限定名
  3. 每个自动配置类通过 @Conditional 系列注解判断是否生效：
    • @ConditionalOnClass：classpath 中存在指定类
    • @ConditionalOnMissingBean：容器中不存在指定 Bean
    • @ConditionalOnProperty：配置属性满足条件
  4. 生效的配置类向容器注册 Bean
// 自定义 Starter 示例
@Configuration
@ConditionalOnClass(RedisTemplate.class)
@EnableConfigurationProperties(RedisProperties.class)
public class RedisAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public RedisTemplate<String, Object> redisTemplate(
            RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        return template;
    }
}

Spring Boot 启动流程

  1. new SpringApplication() → 推断应用类型、加载 SpringApplicationRunListener
  2. 准备 Environment（配置文件加载）
  3. 创建 ApplicationContext
  4. prepareContext → 加载 BeanDefinition
  5. refreshContext → 核心 refresh() 方法（Bean 实例化、AOP、自动配置生效）
  6. afterRefresh → 调用 Runner（CommandLineRunner / ApplicationRunner）

🌐 Spring MVC

请求处理流程

  1. 客户端发送请求 → DispatcherServlet（前端控制器）
  2. DispatcherServlet 调用 HandlerMapping 找到对应的 Handler（Controller 方法）
  3. 通过 HandlerAdapter 执行 Handler
  4. Handler 返回 ModelAndView（或 @ResponseBody 直接返回 JSON）
  5. ViewResolver 解析视图（前后端分离场景不需要）
  6. 返回响应给客户端

全局异常处理

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusiness(BusinessException e) {
        log.warn("Business error: {}", e.getMessage());
        return Result.fail(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<?> handleValidation(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining("; "));
        return Result.fail(400, msg);
    }

    @ExceptionHandler(Exception.class)
    public Result<?> handleException(Exception e) {
        log.error("Unknown error", e);
        return Result.fail(500, "系统繁忙，请稍后重试");
    }
}

Nacos 2.x 深度解析（重点）

架构升级：HTTP → gRPC

Nacos 2.x 引入 gRPC 通信，解决 1.x 的性能瓶颈：

  • 连接层：gRPC 长连接替代 HTTP 短连接，减少连接开销
  • 服务注册：gRPC 双向流，注册请求秒级生效（1.x 有延迟）
  • 配置推送：gRPC Server Push 替代 HTTP 长轮询，推送延迟从秒级降到毫秒级
  • 协议兼容：2.x 兼容 1.x 的 OpenAPI，平滑升级

AP/CP 模式选择

  • 临时实例（AP·Distro 协议）：默认模式，客户端心跳保活，不健康自动剔除。适合大多数场景
  • 持久实例（CP·Raft 协议）：服务端主动探测，需要手动注册/注销。适合数据库、中间件等基础设施

Nacos 面试高频问题

Nacos 如何保证服务发现的实时性？▼
Nacos 2.x 采用 Push + Pull 结合：① gRPC 长连接实现服务端主动推送（毫秒级）；② 客户端定时拉取作为兜底（默认 10 秒）；③ 服务端有变更时通过 UDP 推送通知客户端。三者结合保证实时性和可靠性。
Nacos 和 Eureka 的核心区别？▼
  • CP/AP：Eureka 只支持 AP，Nacos 支持 AP + CP 切换
  • 配置中心：Nacos 内置配置中心，Eureka 需要配合 Spring Cloud Config
  • 通信协议：Nacos 2.x 用 gRPC，Eureka 用 HTTP
  • 健康检查：Nacos 支持服务端探测（TCP/HTTP），Eureka 只有客户端心跳
  • 维护状态：Eureka 2.x 已停止开发，Nacos 持续迭代

Spring Cloud Gateway 核心概念

  • Route（路由）：网关的基本单元，由 ID、URI、Predicate、Filter 组成
  • Predicate（断言）：匹配请求条件（Path、Method、Header、Query 等）
  • Filter（过滤器）：对请求/响应进行处理（添加 Header、限流、鉴权等）

Gateway 完整配置示例

# Spring Cloud Gateway 完整配置
spring:
  cloud:
    gateway:
      # 全局跨域配置
      globalcors:
        cors-configurations:
          '[/**]':
            allowedOriginPatterns: "*"
            allowedMethods: ["GET", "POST", "PUT", "DELETE"]
            allowCredentials: true

      # 路由配置
      routes:
        # 用户服务
        - id: user-service
          uri: lb://user-service          # lb:// 表示从注册中心负载均衡
          predicates:
            - Path=/api/user/**
            - Method=GET,POST
          filters:
            - StripPrefix=1               # 去掉 /api 前缀
            - name: RequestRateLimiter     # 限流过滤器
              args:
                redis-rate-limiter:
                  replenishRate: 10        # 每秒 10 个请求
                  burstCapacity: 20        # 突发容量 20
                key-resolver: "#{@ipKeyResolver}"

        # 订单服务
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/order/**
          filters:
            - StripPrefix=1
            - name: CircuitBreaker         # 熔断过滤器
              args:
                name: orderCircuitBreaker
                fallbackUri: forward:/fallback/order

        # Nacos 动态路由（推荐）
      discovery:
        locator:
          enabled: true                    # 自动根据 service-id 生成路由
          lower-case-service-id: true      # 服务名小写

Gateway 自定义过滤器

// 全局鉴权过滤器
@Component
public class AuthFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (!StringUtils.hasText(token) || !validateToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        // 将用户信息传递给下游服务
        ServerHttpRequest request = exchange.getRequest().mutate()
            .header("X-User-Id", getUserId(token))
            .build();
        return chain.filter(exchange.mutate().request(request).build());
    }

    @Override
    public int getOrder() { return -1; } // 优先级最高
}

// IP 限流 Key 解析器
@Bean
public KeyResolver ipKeyResolver() {
    return exchange -> Mono.just(
        exchange.getRequest().getRemoteAddress().getAddress().getHostAddress()
    );
}
⚠️ Zuul 1.x 为什么被淘汰？ Zuul 1.x 基于 Servlet 阻塞 I/O，每个请求占用一个线程，并发能力有限（~1k QPS）。 Gateway 基于 WebFlux + Netty 响应式编程，单线程可处理大量请求，性能提升 10 倍以上。 Zuul 2.x 虽然改为非阻塞，但 Netflix 已停止维护，社区不活跃。

OpenFeign 核心用法

// 1. 启用 OpenFeign
@SpringBootApplication
@EnableFeignClients
public class Application {}

// 2. 声明 Feign 客户端
@FeignClient(
    name = "user-service",               // 服务名
    path = "/api/user",
    fallbackFactory = UserClientFallback.class  // 降级工厂
)
public interface UserClient {
    @GetMapping("/{id}")
    Result<User> getUser(@PathVariable("id") Long id);

    @PostMapping
    Result<User> createUser(@RequestBody UserDTO user);
}

// 3. 降级工厂（服务不可用时的兜底逻辑）
@Component
public class UserClientFallback implements FallbackFactory<UserClient> {
    @Override
    public UserClient create(Throwable cause) {
        return new UserClient() {
            @Override
            public Result<User> getUser(Long id) {
                return Result.fail("用户服务不可用，请稍后重试");
            }
        };
    }
}

// 4. 使用
@RestController
public class OrderController {
    @Resource
    private UserClient userClient;

    @GetMapping("/order/{orderId}")
    public Result<OrderVO> getOrder(@PathVariable Long orderId) {
        Order order = orderService.getById(orderId);
        // Feign 调用用户服务
        Result<User> userResult = userClient.getUser(order.getUserId());
        return Result.ok(buildOrderVO(order, userResult.getData()));
    }
}

Sentinel 注解实战

// @SentinelResource 注解方式
@SentinelResource(
    value = "getOrder",
    blockHandler = "getOrderBlock",   // 限流/降级处理
    fallback = "getOrderFallback",    // 业务异常兜底
    exceptionsToIgnore = BusinessException.class
)
public Order getOrder(Long id) {
    return orderMapper.selectById(id);
}

// 限流/降级处理方法（参数必须与原方法一致，加 BlockException）
public Order getOrderBlock(Long id, BlockException ex) {
    log.warn("限流: {}", ex.getClass().getSimpleName());
    return Order.defaultOrder();
}

// 业务异常兜底
public Order getOrderFallback(Long id, Throwable t) {
    log.warn("降级: {}", t.getMessage());
    return Order.defaultOrder();
}

Resilience4j 核心用法（国际化方案）

// Resilience4j 轻量级熔断器
// 1. 断路器（CircuitBreaker）
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)           // 失败率 50% 触发熔断
    .waitDurationInOpenState(Duration.ofSeconds(10))  // 熔断 10 秒
    .slidingWindowSize(10)              // 滑动窗口大小
    .build();

CircuitBreaker cb = CircuitBreaker.of("userService", config);

// 2. 组合使用：断路器 + 限流 + 重试
Supplier<Order> decoratedSupplier = Decorators.ofSupplier(() -> getOrder(id))
    .withCircuitBreaker(cb)
    .withRateLimiter(rateLimiter)
    .withRetry(retry)
    .decorate();

// 3. Spring Boot 注解方式
@CircuitBreaker(name = "userService", fallbackMethod = "fallback")
@RateLimiter(name = "userService")
@Retry(name = "userService")
public Order getOrder(Long id) { /* ... */ }

AT 模式原理（核心）

AT 模式执行流程（两阶段）：

第一阶段（业务执行）：
1. 解析 SQL，生成 before image（执行前数据快照）
2. 执行业务 SQL
3. 生成 after image（执行后数据快照）
4. before/after image 写入 undo_log 表（本地事务）
5. 注册分支事务到 TC（Transaction Coordinator）
6. 提交本地事务，释放锁

第二阶段 - 提交（正常情况）：
1. TC 通知所有分支：全局提交
2. 分支只需删除 undo_log（异步，性能好）
3. 结束

第二阶段 - 回滚（异常情况）：
1. TC 通知所有分支：全局回滚
2. 分支读取 undo_log 中的 before image
3. 校验当前数据是否与 after image 一致（防脏写）
4. 用 before image 生成反向 SQL 执行回滚
5. 删除 undo_log

Seata AT 实战配置

// 1. 全局事务注解
@GlobalTransactional(timeoutMills = 30000, name = "create-order")
public void createOrder(OrderDTO dto) {
    // 1. 扣减库存（远程调用，分支事务）
    storageClient.deduct(dto.getItemId(), dto.getCount());
    // 2. 扣减余额（远程调用，分支事务）
    accountClient.debit(dto.getUserId(), dto.getTotalAmount());
    // 3. 创建订单（本地事务，分支事务）
    orderMapper.insert(buildOrder(dto));
    // 任一分支异常 → 全局回滚
}
# Seata 配置
seata:
  application-id: ${spring.application.name}
  tx-service-group: my_tx_group           # 事务组
  registry:
    type: nacos                            # 注册中心
    nacos:
      server-addr: 127.0.0.1:8848
  config:
    type: nacos                            # 配置中心
    nacos:
      server-addr: 127.0.0.1:8848
  service:
    vgroup-mapping:
      my_tx_group: default                 # 事务组映射到集群
⚠️ AT 模式注意事项 ① 必须有主键（undo_log 需要）；② 不支持 SELECT FOR UPDATE（需要手动处理）；③ 全局锁会导致性能下降，大事务慎用；④ 多数据源场景需保证 undo_log 与业务数据在同一库。

💳 Spring 事务管理

事务传播行为（7 种）

传播行为说明
REQUIRED（默认）有事务就加入，没有就新建
REQUIRES_NEW始终新建事务，挂起当前事务
SUPPORTS有事务就加入，没有就非事务执行
NOT_SUPPORTED非事务执行，挂起当前事务
MANDATORY必须在事务中，否则抛异常
NEVER不能在事务中，否则抛异常
NESTED有事务就创建嵌套事务（Savepoint），没有就新建

事务失效的 8 种场景（高频）

  1. 方法非 public：Spring AOP 只能代理 public 方法
  2. 自调用：同一个类中 A 方法调 B 方法，B 的 @Transactional 不生效（因为没走代理）
  3. 异常被 catch：吞掉异常后 Spring 感知不到，不会回滚
  4. 抛出非 RuntimeException：默认只回滚 RuntimeException 和 Error
  5. propagation 设置错误：如 NOT_SUPPORTED 以非事务运行
  6. rollbackFor 未指定：需要回滚 checked 异常时需显式指定 @Transactional(rollbackFor = Exception.class)
  7. 数据库引擎不支持：MySQL 的 MyISAM 不支持事务
  8. 多线程调用：新线程中的操作不在同一个事务中
// ❌ 自调用导致事务失效
@Service
public class UserService {
    public void createUser() {
        // 业务逻辑...
        this.updateUser(); // 直接调用，不走代理，事务失效！
    }

    @Transactional
    public void updateUser() { // 事务不生效
        // ...
    }
}

// ✅ 解决方案：注入自身代理对象
@Service
public class UserService {
    @Resource
    private UserService self; // 注入代理对象

    public void createUser() {
        self.updateUser(); // 走代理，事务生效
    }
}
🚫 事务最佳实践 ① 始终指定 rollbackFor = Exception.class；② 事务方法尽量小，不要包含远程调用；③ 避免大事务（长时间持有数据库连接）。

🔄 IoC 与依赖注入

IoC 容器原理

IoC（控制反转）是 Spring 的核心思想，将对象的创建和依赖管理交给 Spring 容器。核心接口：

  • BeanFactory：最基础的容器接口，懒加载 Bean
  • ApplicationContext：BeanFactory 的子接口，功能更丰富（国际化、事件发布、AOP 等）

Bean 生命周期（面试高频）

  1. 实例化：通过构造器或工厂方法创建 Bean 实例
  2. 属性赋值：注入依赖（@Autowired、@Value 等）
  3. Aware 回调：BeanNameAware → BeanFactoryAware → ApplicationContextAware
  4. BeanPostProcessor#postProcessBeforeInitialization
  5. 初始化：@PostConstruct → InitializingBean#afterPropertiesSet → 自定义 init-method
  6. BeanPostProcessor#postProcessAfterInitialization（AOP 代理在此阶段创建）
  7. 使用 Bean
  8. 销毁：@PreDestroy → DisposableBean#destroy → 自定义 destroy-method

Bean 的作用域

作用域说明
singleton（默认）容器中只有一个实例
prototype每次获取都创建新实例
request每个 HTTP 请求一个实例（Web）
session每个 HTTP Session 一个实例（Web）

循环依赖解决

Spring 通过 三级缓存 解决 singleton 作用域的 setter/field 注入循环依赖：

  • 一级缓存 singletonObjects：完全初始化好的 Bean
  • 二级缓存 earlySingletonObjects：提前曝光的 Bean（可能是代理对象）
  • 三级缓存 singletonFactories：Bean 工厂（用于创建早期引用）
⚠️ 不能解决的循环依赖 构造器注入的循环依赖无法解决（因为对象还没创建完）。Spring Boot 2.6+ 默认禁止循环依赖。

✂️ AOP 面向切面编程

AOP 核心概念

  • 切面（Aspect）：横切关注点的模块化（如日志、事务）
  • 连接点（JoinPoint）：程序执行的某个位置（Spring AOP 只支持方法级别的连接点）
  • 切入点（Pointcut）：匹配连接点的表达式
  • 通知（Advice）：在切入点执行的动作（Before、After、Around、AfterReturning、AfterThrowing）

AOP 实现原理

  • JDK 动态代理：基于接口，通过 Proxy.newProxyInstance() 生成代理类
  • CGLIB 代理：基于继承，通过字节码生成目标类的子类
  • Spring 默认：有接口用 JDK 代理，无接口用 CGLIB（Spring Boot 2.x 默认全部使用 CGLIB）
// AOP 实战：统一日志切面
@Aspect
@Component
public class LogAspect {

    @Around("execution(* com.example.service.*.*(..))")
    public Object log(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed();
            long cost = System.currentTimeMillis() - start;
            log.info("{}.{} cost {}ms",
                pjp.getTarget().getClass().getSimpleName(),
                pjp.getSignature().getName(), cost);
            return result;
        } catch (Throwable e) {
            log.error("Method error: {}", e.getMessage());
            throw e;
        }
    }
}

🚀 Spring Boot

自动配置原理（核心面试题）

Spring Boot 自动配置的入口是 @SpringBootApplication 注解，它组合了：

  • @SpringBootConfiguration：本质是 @Configuration
  • @EnableAutoConfiguration：开启自动配置
  • @ComponentScan：包扫描

自动配置流程

  1. @EnableAutoConfiguration 通过 @Import(AutoConfigurationImportSelector.class) 导入配置
  2. AutoConfigurationImportSelector 读取 META-INF/spring.factories（或 Spring Boot 3.x 的 AutoConfiguration.imports）中的自动配置类全限定名
  3. 每个自动配置类通过 @Conditional 系列注解判断是否生效：
    • @ConditionalOnClass：classpath 中存在指定类
    • @ConditionalOnMissingBean：容器中不存在指定 Bean
    • @ConditionalOnProperty：配置属性满足条件
  4. 生效的配置类向容器注册 Bean
// 自定义 Starter 示例
@Configuration
@ConditionalOnClass(RedisTemplate.class)
@EnableConfigurationProperties(RedisProperties.class)
public class RedisAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public RedisTemplate<String, Object> redisTemplate(
            RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        return template;
    }
}

Spring Boot 启动流程

  1. new SpringApplication() → 推断应用类型、加载 SpringApplicationRunListener
  2. 准备 Environment（配置文件加载）
  3. 创建 ApplicationContext
  4. prepareContext → 加载 BeanDefinition
  5. refreshContext → 核心 refresh() 方法（Bean 实例化、AOP、自动配置生效）
  6. afterRefresh → 调用 Runner（CommandLineRunner / ApplicationRunner）

🌐 Spring MVC

请求处理流程

  1. 客户端发送请求 → DispatcherServlet（前端控制器）
  2. DispatcherServlet 调用 HandlerMapping 找到对应的 Handler（Controller 方法）
  3. 通过 HandlerAdapter 执行 Handler
  4. Handler 返回 ModelAndView（或 @ResponseBody 直接返回 JSON）
  5. ViewResolver 解析视图（前后端分离场景不需要）
  6. 返回响应给客户端

常用注解

注解作用
@RestController= @Controller + @ResponseBody
@RequestMapping映射请求路径和方法
@PathVariableURL 路径参数 /users/{id}
@RequestParam查询参数 ?name=xxx
@RequestBody请求体 JSON 反序列化
@ExceptionHandler局部异常处理
@RestControllerAdvice全局异常处理 + 数据绑定

全局异常处理

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusiness(BusinessException e) {
        log.warn("Business error: {}", e.getMessage());
        return Result.fail(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<?> handleValidation(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining("; "));
        return Result.fail(400, msg);
    }

    @ExceptionHandler(Exception.class)
    public Result<?> handleException(Exception e) {
        log.error("Unknown error", e);
        return Result.fail(500, "系统繁忙，请稍后重试");
    }
}

Nacos 注册中心原理

  • 服务注册：服务启动时向 Nacos 发送注册请求（HTTP/gRPC），携带服务名、IP、端口、元数据
  • 心跳检测：客户端每 5 秒发送心跳，15 秒未收到心跳标记为不健康，30 秒剔除
  • 服务发现：客户端定时拉取服务列表（Pull 模式），同时 Nacos 主动推送变更（Push 模式）
  • AP/CP 切换：临时实例用 Distro 协议（AP），持久实例用 Raft 协议（CP）

Sentinel 熔断降级

Sentinel 提供丰富的流控和熔断策略：

  • 流控规则：QPS 限流、线程数限流、关联限流、链路限流
  • 熔断策略：慢调用比例、异常比例、异常数
  • 热点参数限流：针对特定参数值限流
  • 集群限流：多节点共享限流计数

💳 Spring 事务管理

事务传播行为（7 种）

传播行为说明
REQUIRED（默认）有事务就加入，没有就新建
REQUIRES_NEW始终新建事务，挂起当前事务
SUPPORTS有事务就加入，没有就非事务执行
NOT_SUPPORTED非事务执行，挂起当前事务
MANDATORY必须在事务中，否则抛异常
NEVER不能在事务中，否则抛异常
NESTED有事务就创建嵌套事务（Savepoint），没有就新建

事务失效的 8 种场景（高频）

  1. 方法非 public：Spring AOP 只能代理 public 方法
  2. 自调用：同一个类中 A 方法调 B 方法，B 的 @Transactional 不生效（因为没走代理）
  3. 异常被 catch：吞掉异常后 Spring 感知不到，不会回滚
  4. 抛出非 RuntimeException：默认只回滚 RuntimeException 和 Error
  5. propagation 设置错误：如 NOT_SUPPORTED 以非事务运行
  6. rollbackFor 未指定：需要回滚 checked 异常时需显式指定 @Transactional(rollbackFor = Exception.class)
  7. 数据库引擎不支持：MySQL 的 MyISAM 不支持事务
  8. 多线程调用：新线程中的操作不在同一个事务中
// ❌ 自调用导致事务失效
@Service
public class UserService {
    public void createUser() {
        // 业务逻辑...
        this.updateUser(); // 直接调用，不走代理，事务失效！
    }

    @Transactional
    public void updateUser() { // 事务不生效
        // ...
    }
}

// ✅ 解决方案：注入自身代理对象
@Service
public class UserService {
    @Resource
    private UserService self; // 注入代理对象

    public void createUser() {
        self.updateUser(); // 走代理，事务生效
    }
}
🚫 事务最佳实践 ① 始终指定 rollbackFor = Exception.class；② 事务方法尽量小，不要包含远程调用；③ 避免大事务（长时间持有数据库连接）。

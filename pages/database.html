🔍 MySQL 索引

B+ 树原理

InnoDB 使用 B+ 树作为索引结构，理解 B+ 树是理解索引优化的基础：

  • 非叶子节点只存 key（索引值），不存 data，因此单个节点能存更多 key，树更矮
  • 叶子节点存放完整数据，且通过双向链表相连，支持范围查询
  • 树高通常 2-4 层：千万级数据只需 2-3 次磁盘 IO

聚簇索引 vs 非聚簇索引

类型说明特点
聚簇索引（主键索引）叶子节点存完整行数据一张表只有一个，主键即数据
非聚簇索引（二级索引）叶子节点存主键值需要回表查询完整数据

覆盖索引

当查询的字段全部在索引中时，无需回表，称为覆盖索引。EXPLAIN 中 Extra 显示 Using index。

-- 联合索引 (name, age)
SELECT name, age FROM user WHERE name = '张三';
-- ✅ 覆盖索引，不需要回表

SELECT name, age, email FROM user WHERE name = '张三';
-- ❌ email 不在索引中，需要回表

联合索引与最左前缀原则

联合索引 (a, b, c) 相当于创建了 (a)、(a, b)、(a, b, c) 三个索引。

-- 索引 (a, b, c)
WHERE a = 1 AND b = 2 AND c = 3  -- ✅ 全部命中
WHERE a = 1 AND b = 2             -- ✅ 命中 a, b
WHERE a = 1                        -- ✅ 命中 a
WHERE b = 2 AND c = 3             -- ❌ 跳过 a，索引失效
WHERE a = 1 AND c = 3             -- ⚠️ 只命中 a，c 无法使用索引

索引失效的常见场景

  1. 对索引列使用函数或运算：WHERE YEAR(create_time) = 2024
  2. 隐式类型转换：WHERE phone = 13800000000（phone 是 varchar）
  3. LIKE 以通配符开头：WHERE name LIKE '%张'
  4. OR 条件中有非索引列
  5. 违反最左前缀原则
  6. 使用 != 或 NOT IN（可能不走索引，取决于数据分布）

🔒 MySQL 事务

ACID 特性

  • 原子性（Atomicity）：undo log 实现回滚
  • 一致性（Consistency）：事务前后数据完整性不变
  • 隔离性（Isolation）：MVCC + 锁机制实现
  • 持久性（Durability）：redo log 实现崩溃恢复

隔离级别

隔离级别脏读不可重复读幻读
READ UNCOMMITTED✅✅✅
READ COMMITTED（Oracle 默认）❌✅✅
REPEATABLE READ（MySQL 默认）❌❌部分解决 ✅
SERIALIZABLE❌❌❌

MVCC 原理（高频）

MVCC（多版本并发控制）通过 Read View 和 undo log 版本链实现无锁读：

  • 隐藏字段：每行数据有 DB_TRX_ID（最后修改的事务ID）和 DB_ROLL_PTR（回滚指针，指向 undo log）
  • undo log 版本链：通过 ROLL_PTR 串成链表，记录数据的历史版本
  • Read View：事务开始时创建，包含活跃事务ID列表，用于判断哪个版本可见

RC vs RR 的区别：

  • READ COMMITTED：每次 SELECT 都创建新的 Read View
  • REPEATABLE READ：只在事务第一次 SELECT 时创建 Read View，之后复用

InnoDB 锁机制

锁类型粒度说明
共享锁（S Lock）行级读锁，多个事务可共享
排他锁（X Lock）行级写锁，互斥
意向锁（IS/IX）表级标识事务意图，加速锁冲突判断
间隙锁（Gap Lock）行间RR 级别下防止幻读
临键锁（Next-Key Lock）行+间隙= Record Lock + Gap Lock

⚡ 查询优化

EXPLAIN 分析

EXPLAIN SELECT * FROM user WHERE name = '张三';
字段关注点
type访问类型：ALL(全表扫描) → index → range → ref → eq_ref → const
key实际使用的索引
rows预估扫描行数
ExtraUsing index(覆盖索引)、Using filesort(文件排序)、Using temporary(临时表)

慢查询优化思路

  1. 开启慢查询日志定位问题 SQL
  2. EXPLAIN 分析执行计划
  3. 优化方向：
    • 添加合适的索引（覆盖索引优先）
    • 避免 SELECT *，只查需要的字段
    • 避免索引失效的写法
    • 大分页优化：SELECT * FROM t WHERE id > #{lastId} LIMIT 20（游标分页）
    • JOIN 优化：小表驱动大表，被驱动表的关联字段加索引

分库分表

当单表数据量超过千万级别时考虑分表：

  • 垂直分库：按业务拆分（用户库、订单库、商品库）
  • 水平分表：按分片键（如 user_id % N）将数据分散到多张表
  • 常用中间件：ShardingSphere、MyCat
  • 分片策略：取模、范围、一致性 Hash

跳表（SkipList）原理

ZSet 底层使用跳表（而非红黑树），原因是：

  • 跳表实现简单，范围查询高效（直接遍历底层链表）
  • 插入/删除时只需调整局部指针，不像红黑树需要全局平衡
  • 跳表的查找时间复杂度 O(log n)，与平衡树相当

缓存三大问题

缓存穿透

查询不存在的数据，缓存和数据库都查不到，每次都打到数据库。

  • 布隆过滤器：在缓存前加一层布隆过滤器，快速判断 key 是否可能存在
  • 缓存空值：查询结果为空时也缓存，设置较短过期时间

缓存击穿

热点 key 过期瞬间，大量请求打到数据库。

  • 互斥锁：只允许一个线程去数据库加载，其他线程等待
  • 逻辑过期：不设置 TTL，在 value 中存逻辑过期时间，过期后异步更新

缓存雪崩

大量 key 同时过期或 Redis 宕机，请求全部打到数据库。

  • 过期时间加随机值：避免同时过期
  • 集群高可用：Redis Cluster / Sentinel
  • 限流降级：对数据库请求限流

缓存与数据库一致性

推荐方案：Cache Aside Pattern

  1. 读：先读缓存，miss 则读数据库并写入缓存
  2. 写：先更新数据库，再删除缓存
⚠️ 延迟双删 先删缓存 → 更新数据库 → 延迟一段时间 → 再删缓存。应对"读请求在写请求更新数据库后、删缓存前把旧值写入缓存"的并发问题。

🛡️ Redis 高可用

持久化

方式原理优缺点
RDB（快照）fork 子进程，将内存数据序列化到 dump.rdb恢复快，但可能丢失最后一次快照后的数据
AOF（追加日志）记录每条写命令到 appendonly.aof数据更安全（everysec），但文件大、恢复慢
混合持久化RDB + 增量 AOFRedis 4.0+ 推荐，兼顾恢复速度和数据安全

主从复制

  • 全量同步：首次连接，Master 生成 RDB 发送给 Slave
  • 增量同步：断线重连后，基于 repl_backlog 做增量同步

Sentinel 哨兵

自动监控和故障转移：

  1. 每个 Sentinel 每秒向 Master 发送 PING，主观下线（SDOWN）
  2. 当 quorum 个 Sentinel 都认为 Master 下线 → 客观下线（ODOWN）
  3. Sentinel 选举 Leader，由 Leader 执行故障转移（选新 Master）

Redis Cluster

  • 数据分片：16384 个 slot，CRC16(key) % 16384 定位 slot
  • 去中心化：每个节点都能处理请求，不在自己 slot 范围则 MOVED 重定向
  • 扩缩容：添加/移除节点时迁移 slot
  • 故障转移：节点间互相 PING，超过半数认为下线则触发选举

分布式锁实现

// Redisson 分布式锁（推荐）
@Resource
private RedissonClient redissonClient;

public void deductStock() {
    RLock lock = redissonClient.getLock("stock:lock");
    try {
        // 尝试加锁，最多等待 3 秒，10 秒自动释放
        if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
            // 业务逻辑
        }
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}

🗃️ MyBatis 深入

架构与执行流程

MyBatis 是一个半自动 ORM 框架，开发者手写 SQL，MyBatis 负责参数映射和结果集映射。

核心组件：

  • SqlSessionFactory：全局单例，负责创建 SqlSession（通过 SqlSessionFactoryBuilder 解析 XML/注解配置）
  • SqlSession：一次数据库会话，非线程安全，每次请求创建新实例
  • Executor：SQL 执行器（Simple、Reuse、Batch 三种类型）
  • MappedStatement：封装一条 SQL 的所有信息（id、SQL 文本、参数类型、结果类型）
  • StatementHandler：处理 JDBC Statement 的创建和参数设置
  • ParameterHandler：将用户传入的参数设置到 PreparedStatement
  • ResultSetHandler：将 JDBC ResultSet 转换为 Java 对象

执行流程：

  1. 加载配置文件（mybatis-config.xml + Mapper.xml），构建 Configuration 对象
  2. 创建 SqlSessionFactory → 打开 SqlSession
  3. 通过 Mapper 接口的代理对象（MapperProxy）调用方法
  4. Executor 查询时先检查二级缓存 → 一级缓存 → 查询数据库
  5. StatementHandler 创建 PreparedStatement，ParameterHandler 设置参数
  6. 执行 SQL，ResultSetHandler 将结果映射为 Java 对象返回

#{} vs ${} 区别

对比项#{}${}
处理方式预编译，使用 ? 占位符字符串拼接，直接替换
SQL 注入✅ 安全，防止 SQL 注入❌ 不安全，有 SQL 注入风险
使用场景参数值传递（WHERE、INSERT 值）动态表名、列名、ORDER BY
JDBC 对应PreparedStatementStatement
<!-- #{} 预编译，安全 -->
<select id="getUserById" resultType="User">
    SELECT * FROM user WHERE id = #{id}
    <!-- 最终执行：SELECT * FROM user WHERE id = ? -->
</select>

<!-- ${} 字符串拼接，仅用于动态列名/表名 -->
<select id="selectByColumn" resultType="User">
    SELECT * FROM user ORDER BY ${columnName} ${order}
    <!-- 最终执行：SELECT * FROM user ORDER BY create_time DESC -->
</select>
⚠️ 安全提醒 使用 ${} 时必须在代码层做白名单校验，防止 SQL 注入。例如限制 columnName 只能是已知的列名。

动态 SQL

MyBatis 提供强大的动态 SQL 标签，避免手动拼接 SQL 字符串：

<!-- if - 条件判断 -->
<select id="findUsers" resultType="User">
    SELECT * FROM user WHERE 1=1
    <if test="name != null and name != ''">
        AND name LIKE CONCAT('%', #{name}, '%')
    </if>
    <if test="age != null">
        AND age = #{age}
    </if>
</select>

<!-- where - 自动去除前导 AND/OR -->
<select id="findUsers" resultType="User">
    SELECT * FROM user
    <where>
        <if test="name != null"> AND name = #{name} </if>
        <if test="age != null"> AND age = #{age} </if>
    </where>
</select>

<!-- foreach - 遍历集合（批量查询/插入） -->
<select id="findByIds" resultType="User">
    SELECT * FROM user WHERE id IN
    <foreach collection="ids" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</select>

<!-- choose/when/otherwise - 类似 switch-case -->
<select id="findUser" resultType="User">
    SELECT * FROM user
    <where>
        <choose>
            <when test="id != null"> id = #{id} </when>
            <when test="name != null"> name = #{name} </when>
            <otherwise> 1 = 1 </otherwise>
        </choose>
    </where>
</select>

<!-- set - UPDATE 语句自动去除末尾逗号 -->
<update id="updateUser">
    UPDATE user
    <set>
        <if test="name != null"> name = #{name}, </if>
        <if test="age != null"> age = #{age}, </if>
    </set>
    WHERE id = #{id}
</update>

<!-- sql/include - SQL 片段复用 -->
<sql id="baseColumns"> id, name, age, email </sql>
<select id="findAll" resultType="User">
    SELECT <include refid="baseColumns"/> FROM user
</select>

一级缓存与二级缓存

对比项一级缓存二级缓存
作用域SqlSession 级别Mapper（namespace）级别
默认状态默认开启默认关闭，需手动配置
生命周期SqlSession 关闭即失效跨 SqlSession 共享
失效条件执行增删改、手动 clearCache()执行增删改（整个 namespace 失效）
实现原理HashMap（BaseExecutor 内部）Decorator 模式（CachingExecutor）
注意事项Spring 中每个请求通常一个 SqlSession，一级缓存作用有限分布式环境下容易出现脏数据，生产慎用
<!-- 开启二级缓存（Mapper.xml 中配置） -->
<cache
    eviction="LRU"           <!-- 淘汰策略：LRU/FIFO/SOFT/WEAK -->
    flushInterval="60000"    <!-- 刷新间隔（毫秒） -->
    size="1024"              <!-- 缓存对象数量 -->
    readOnly="true"          <!-- 只读，返回同一引用（性能好） -->
/>
⚠️ 生产环境建议 分布式环境下 MyBatis 二级缓存容易导致脏数据（各节点缓存不一致），生产环境推荐使用 Redis 作为缓存层，禁用 MyBatis 二级缓存。

插件机制（Interceptor）

MyBatis 允许在四大对象的调用链路上插入自定义逻辑（AOP 思想）：

  • Executor：执行器（query、update、commit 等）
  • StatementHandler：Statement 处理器（prepare、parameterize、batch 等）
  • ParameterHandler：参数处理器（setParameters）
  • ResultSetHandler：结果集处理器（handleResultSets）
// 自定义分页插件示例
@Intercepts({
    @Signature(type = StatementHandler.class,
                method = "prepare",
                args = {Connection.class, Integer.class})
})
public class PageInterceptor implements Interceptor {

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        StatementHandler handler = (StatementHandler) invocation.getTarget();
        BoundSql boundSql = handler.getBoundSql();
        String originalSql = boundSql.getSql();

        // 改写 SQL，添加 LIMIT
        String pageSql = originalSql + " LIMIT " + offset + ", " + pageSize;
        // 通过反射替换 SQL
        Field sqlField = BoundSql.class.getDeclaredField("sql");
        sqlField.setAccessible(true);
        sqlField.set(boundSql, pageSql);

        return invocation.proceed();
    }

    @Override
    public Object plugin(Object target) {
        return Plugin.wrap(target, this);
    }
}

MyBatis 高频面试题

MyBatis 中 #{} 和 ${} 的区别？ 高频 ▼

#{} 使用预编译（PreparedStatement），参数用 ? 占位，安全防注入。${} 直接字符串拼接，有 SQL 注入风险。

#{} 适用于参数值，${} 仅用于动态表名、列名、ORDER BY 等不能用占位符的场景，使用时需做白名单校验。

MyBatis 的一级缓存和二级缓存有什么区别？ 高频 ▼

一级缓存是 SqlSession 级别，默认开启，同一 SqlSession 内相同查询直接返回缓存结果，执行增删改后自动失效。

二级缓存是 Mapper 级别，默认关闭，跨 SqlSession 共享。需在 Mapper.xml 中配置 <cache/>，实体类实现 Serializable。分布式环境下容易出现脏数据，生产推荐用 Redis 替代。

MyBatis 的动态 SQL 有哪些标签？ 高频 ▼
  • <if>：条件判断
  • <where>：自动去除前导 AND/OR
  • <set>：UPDATE 时自动去除末尾逗号
  • <choose>/<when>/<otherwise>：类似 switch-case
  • <foreach>：遍历集合，常用于 IN 查询和批量插入
  • <trim>：自定义前缀/后缀的添加和去除
  • <sql>/<include>：SQL 片段复用
MyBatis 的 Mapper 接口工作原理是什么？ 高频 ▼

MyBatis 启动时为每个 Mapper 接口生成一个 MapperProxy 代理对象（JDK 动态代理），存入 Configuration 的 Map 中。

调用 Mapper 方法时，MapperProxy 拦截方法调用，根据接口全限定名 + 方法名找到对应的 MappedStatement，然后委托给 SqlSession 执行 SQL。

这也是为什么 Mapper 接口不需要实现类的原因——由动态代理在运行时生成。

MyBatis-Plus 相比原生 MyBatis 有哪些优势？ 中频 ▼
  • 通用 CRUD：继承 BaseMapper 即可获得单表增删改查，无需写 XML
  • 条件构造器：LambdaQueryWrapper 链式构建查询条件，类型安全
  • 分页插件：内置 PaginationInnerInterceptor，开箱即用
  • 代码生成器：根据数据库表自动生成 Entity、Mapper、Service、Controller
  • 逻辑删除：配置后自动在 SQL 中添加 deleted 条件
  • 自动填充：createTime、updateTime 等字段自动填充
  • 乐观锁：@Version 注解支持

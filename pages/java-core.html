📦 集合框架

HashMap 底层原理（高频）

HashMap 是 Java 中最常用的集合之一，面试必问。核心要点：

数据结构

JDK 8 中 HashMap 采用 数组 + 链表 + 红黑树 结构。当链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转为红黑树；当红黑树节点 ≤ 6 时退化为链表。

// HashMap 核心参数
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4;   // 16
static final float DEFAULT_LOAD_FACTOR = 0.75f;
static final int TREEIFY_THRESHOLD = 8;    // 链表转红黑树阈值
static final int UNTREEIFY_THRESHOLD = 6;  // 红黑树退化阈值
static final int MIN_TREEIFY_CAPACITY = 64; // 树化最小数组长度

put 流程

  1. 计算 key 的 hash 值：(h = key.hashCode()) ^ (h >>> 16) — 高位参与运算减少碰撞
  2. 通过 (n - 1) & hash 定位数组下标
  3. 若桶为空，直接插入新节点
  4. 若桶不为空，遍历链表/红黑树，找到相同 key 则覆盖 value
  5. 若无相同 key，尾插法插入新节点（JDK 7 是头插法，多线程下会形成环）
  6. 若节点数超过阈值（capacity × loadFactor），进行 2 倍扩容
💡 面试要点 JDK 7 的 HashMap 在多线程扩容时，头插法可能导致链表成环（死循环）。JDK 8 改用尾插法解决了这个问题，但 HashMap 本身仍然不是线程安全的。

为什么 HashMap 容量总是 2 的幂？

因为定位桶的运算 (n - 1) & hash 等价于 hash % n（当 n 为 2 的幂时），位运算效率远高于取模。同时保证 hash 值的低位充分散列。

ConcurrentHashMap 原理

JDK 8 中 ConcurrentHashMap 放弃了分段锁（Segment），采用 CAS + synchronized 实现更细粒度的并发控制：

  • put 操作：对桶的头节点加 synchronized 锁，锁粒度从段级别降到桶级别
  • size() 方法：使用 baseCount + CounterCell[] 分散计数，类似 LongAdder 思路
  • 扩容：多线程协助扩容（transfer），每个线程负责一段桶的迁移
// ConcurrentHashMap put 核心逻辑简化
final V putVal(K key, V value) {
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        if (tab == null || (n = tab.length) == 0)

                break;  // CAS 插入成功
        } else {
            synchronized (f) {  // 锁住桶头节点
                // 链表/红黑树插入逻辑...
            }
        }

}

ArrayList vs LinkedList

特性ArrayListLinkedList
底层结构动态数组双向链表
随机访问O(1) ✅O(n) ❌
头部插入O(n)（需移动元素）O(1) ✅
尾部插入均摊 O(1)O(1)
内存占用较少（连续空间）较多（Node 指针开销）
扩容1.5 倍扩容，涉及数组拷贝无需扩容
💡 实际开发建议 绝大多数场景优先使用 ArrayList，CPU 缓存友好性远超 LinkedList。LinkedList 仅在频繁头部插入/删除且不需要随机访问时考虑。

TreeMap 与 LinkedHashMap

  • TreeMap：基于红黑树，key 有序，支持自定义 Comparator，查找/插入/删除均为 O(log n)
  • LinkedHashMap：HashMap + 双向链表，维护插入顺序（或访问顺序），可用于实现 LRU 缓存
// 用 LinkedHashMap 实现简单 LRU 缓存
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxSize;

    public LRU(int maxSize) {
        super(16, 0.75f, true); // accessOrder = true
        this.maxSize = maxSize;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxSize; // 超过容量自动淘汰最久未访问的
    }
}

⚡ 并发编程

线程池原理（必问）

Java 线程池通过 ThreadPoolExecutor 实现，核心参数理解是面试关键：

public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数（常驻线程）
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 非核心线程存活时间
    TimeUnit unit,         // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory factory,              // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)

任务提交流程

  1. 当前线程数 < corePoolSize → 创建核心线程执行任务
  2. 核心线程已满 → 任务放入 workQueue
  3. 队列已满 → 创建非核心线程（不超过 maximumPoolSize）
  4. 线程数已达最大且队列已满 → 执行拒绝策略

四种拒绝策略

策略行为
AbortPolicy（默认）抛出 RejectedExecutionException
CallerRunsPolicy由调用线程执行任务（降级方案）
DiscardPolicy静默丢弃任务
DiscardOldestPolicy丢弃队列最老的任务，重新提交
⚠️ 线程池使用规范 阿里开发手册禁止使用 Executors 创建线程池。newFixedThreadPool 和 newSingleThreadExecutor 的 LinkedBlockingQueue 是无界的，可能导致 OOM；newCachedThreadPool 的 maximumPoolSize 是 Integer.MAX_VALUE，可能创建过多线程。

AQS（AbstractQueuedSynchronizer）原理

AQS 是 Java 并发包的基石，ReentrantLock、Semaphore、CountDownLatch 等都基于它实现。

  • 核心状态：一个 volatile int state 变量（0 表示未锁定，>0 表示锁定/重入次数）
  • CLH 队列：双向链表实现的 FIFO 等待队列
  • 独占模式：tryAcquire → addWaiter → acquireQueued
  • 共享模式：tryAcquireShared（用于 CountDownLatch、Semaphore 等）

volatile 关键字

volatile 保证两个特性：

  1. 可见性：修改后立即刷新到主内存，其他线程读取时从主内存加载
  2. 有序性：禁止指令重排序（通过内存屏障实现）

volatile 不保证原子性，例如 volatile int count; count++ 不是线程安全的，因为 count++ 实际是 read-modify-write 三步操作。

// DCL 双重检查锁单例（volatile 防止指令重排）
public class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                    // 不加 volatile，可能重排序为：
                    // 1. 分配内存 2. 引用赋值 3. 初始化
                    // 其他线程可能拿到未初始化的对象
                }
            }
        }
        return instance;
    }
}

ThreadLocal 原理与内存泄漏

每个 Thread 内部持有一个 ThreadLocalMap，key 是 ThreadLocal 对象（弱引用），value 是存储的值。

  • 内存泄漏原因：ThreadLocal 对象被 GC 后，key 变为 null，但 value 被 Entry 强引用无法回收
  • 解决方案：使用完毕后调用 threadLocal.remove()
  • 典型应用：数据库连接、用户上下文、日期格式化

垃圾回收算法

对象存活判断

  • 引用计数法：每个对象维护引用计数，有循环引用问题（JVM 不使用）
  • 可达性分析：从 GC Roots 出发，不可达的对象判定为可回收

GC Roots 包括：虚拟机栈中引用的对象、静态变量引用的对象、常量引用的对象、JNI 引用的对象等。

回收算法

算法原理优缺点
标记-清除标记可回收对象后清除简单但产生内存碎片
标记-整理标记后将存活对象向一端移动无碎片但移动成本高
复制算法将内存分两块，存活对象复制到另一块无碎片但浪费一半内存
分代收集新生代（复制）+ 老年代（标记整理）JVM 实际使用的策略

类加载机制

类加载过程

加载 → 验证 → 准备 → 解析 → 初始化

双亲委派模型

类加载时先委托父加载器加载，父加载器无法加载时才自己加载。

  • Bootstrap ClassLoader：加载 rt.jar 等核心类库
  • Extension ClassLoader：加载 ext 目录下的类
  • Application ClassLoader：加载 classpath 下的类
💡 打破双亲委派 SPI 机制（ServiceLoader）、OSGi、Tomcat 自定义类加载器等场景会打破双亲委派。典型例子是 JDBC Driver 通过 Thread.currentThread().getContextClassLoader() 加载。

🚀 Java 8-21 新特性

Java 8 核心特性

Lambda 表达式与函数式接口

// 函数式接口：只有一个抽象方法的接口
@FunctionalInterface
interface Converter<F, T> {
    T convert(F from);
}

// Lambda 使用
Converter<String, Integer> converter = Integer::valueOf;
Integer num = converter.convert("123");

// 常用函数式接口
// Predicate<T>    T → boolean    test()
// Function<T,R>   T → R          apply()
// Consumer<T>     T → void       accept()
// Supplier<T>     () → T         get()

Stream API

// Stream 常用操作
List<String> names = Arrays.asList("Alice", "Bob", "Charlie", "David");

// 中间操作（惰性求值）+ 终端操作
List<String> result = names.stream()
    .filter(name -> name.length() > 3)     // 过滤
    .map(String::toUpperCase)               // 映射
    .sorted()                               // 排序
    .collect(Collectors.toList());           // 收集

// 并行流
long count = names.parallelStream()
    .filter(name -> name.startsWith("A"))
    .count();

Optional

// 避免空指针的优雅方式
String name = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");

Java 17+ 重要特性

  • Record（Java 14+）：不可变数据类，自动生成 equals/hashCode/toString
  • Sealed Class（Java 17）：限制哪些类可以继承
  • Pattern Matching（Java 17+）：instanceof 模式匹配
  • Text Block（Java 15）：多行字符串 """..."""
  • Virtual Thread（Java 21）：虚拟线程，高并发场景下大幅减少线程开销
// Record - 不可变数据类
public record Point(int x, int y) {}

// Sealed Class - 限制继承
public sealed class Shape permits Circle, Rectangle, Triangle {}

// Pattern Matching
if (obj instanceof String s && s.length() > 5) {
    System.out.println(s.toUpperCase());
}

// Virtual Thread（Java 21）
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10000; i++) {
        executor.submit(() -> {
            // 每个任务一个虚拟线程，开销极低
            Thread.sleep(Duration.ofSeconds(1));
            return "done";
        });
    }
}

📡 IO 与 NIO

BIO vs NIO vs AIO

模型特点适用场景
BIO（同步阻塞）一个连接一个线程，read 阻塞连接数少且固定的场景
NIO（同步非阻塞）Channel + Buffer + Selector 多路复用高并发连接（聊天、推送）
AIO（异步非阻塞）回调通知，Proactor 模式大量长连接（文件服务器）

NIO 核心组件

  • Channel：双向数据通道（FileChannel、SocketChannel、ServerSocketChannel）
  • Buffer：数据缓冲区（ByteBuffer 最常用），capacity/limit/position 三个核心属性
  • Selector：多路复用器，一个线程管理多个 Channel
💡 Netty Netty 是基于 NIO 的高性能网络框架，面试中常问：线程模型（Reactor 模式）、零拷贝（CompositeByteBuf、FileRegion）、内存管理（池化 ByteBuf）。

🔮 反射与注解

反射原理

Java 反射在运行时获取类的信息并操作类的属性和方法。核心类在 java.lang.reflect 包下：

  • Class<?>：类的元信息
  • Field：成员变量
  • Method：方法
  • Constructor：构造器
// 获取 Class 对象的三种方式
Class<?> c1 = Class.forName("com.example.User");
Class<?> c2 = User.class;
Class<?> c3 = new User().getClass();

// 反射创建实例并调用方法
Object obj = c1.getDeclaredConstructor().newInstance();
Method method = c1.getDeclaredMethod("setName", String.class);
method.setAccessible(true);  // 突破 private 限制
method.invoke(obj, "张三");

注解原理

注解本质是一个继承了 java.lang.annotation.Annotation 的接口。运行时注解通过动态代理实现，编译时注解通过 APT（Annotation Processing Tool）在编译期生成代码。

Spring AOP 底层大量使用了反射和注解，理解反射机制有助于理解框架原理。

📦 集合框架

HashMap 底层原理（高频）

HashMap 是 Java 中最常用的集合之一，面试必问。核心要点：

数据结构

JDK 8 中 HashMap 采用 数组 + 链表 + 红黑树 结构。当链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转为红黑树；当红黑树节点 ≤ 6 时退化为链表。

// HashMap 核心参数
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4;   // 16
static final float DEFAULT_LOAD_FACTOR = 0.75f;
static final int TREEIFY_THRESHOLD = 8;    // 链表转红黑树阈值
static final int UNTREEIFY_THRESHOLD = 6;  // 红黑树退化阈值
static final int MIN_TREEIFY_CAPACITY = 64; // 树化最小数组长度

put 流程

  1. 计算 key 的 hash 值：(h = key.hashCode()) ^ (h >>> 16) — 高位参与运算减少碰撞
  2. 通过 (n - 1) & hash 定位数组下标
  3. 若桶为空，直接插入新节点
  4. 若桶不为空，遍历链表/红黑树，找到相同 key 则覆盖 value
  5. 若无相同 key，尾插法插入新节点（JDK 7 是头插法，多线程下会形成环）
  6. 若节点数超过阈值（capacity × loadFactor），进行 2 倍扩容
💡 面试要点 JDK 7 的 HashMap 在多线程扩容时，头插法可能导致链表成环（死循环）。JDK 8 改用尾插法解决了这个问题，但 HashMap 本身仍然不是线程安全的。

为什么 HashMap 容量总是 2 的幂？

因为定位桶的运算 (n - 1) & hash 等价于 hash % n（当 n 为 2 的幂时），位运算效率远高于取模。同时保证 hash 值的低位充分散列。

ConcurrentHashMap 原理

JDK 8 中 ConcurrentHashMap 放弃了分段锁（Segment），采用 CAS + synchronized 实现更细粒度的并发控制：

  • put 操作：对桶的头节点加 synchronized 锁，锁粒度从段级别降到桶级别
  • size() 方法：使用 baseCount + CounterCell[] 分散计数，类似 LongAdder 思路
  • 扩容：多线程协助扩容（transfer），每个线程负责一段桶的迁移
// ConcurrentHashMap put 核心逻辑简化

TreeMap 与 LinkedHashMap

  • TreeMap：基于红黑树，key 有序，支持自定义 Comparator，查找/插入/删除均为 O(log n)
  • LinkedHashMap：HashMap + 双向链表，维护插入顺序（或访问顺序），可用于实现 LRU 缓存
// 用 LinkedHashMap 实现简单 LRU 缓存
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxSize;

    public LRU(int maxSize) {
        super(16, 0.75f, true); // accessOrder = true
        this.maxSize = maxSize;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxSize; // 超过容量自动淘汰最久未访问的
    }
}

⚡ 并发编程

线程池原理（必问）

Java 线程池通过 ThreadPoolExecutor 实现，核心参数理解是面试关键：

public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数（常驻线程）
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 非核心线程存活时间
    TimeUnit unit,         // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory factory,              // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)

任务提交流程

  1. 当前线程数 < corePoolSize → 创建核心线程执行任务
  2. 核心线程已满 → 任务放入 workQueue
  3. 队列已满 → 创建非核心线程（不超过 maximumPoolSize）
  4. 线程数已达最大且队列已满 → 执行拒绝策略

四种拒绝策略

策略行为
AbortPolicy（默认）抛出 RejectedExecutionException
CallerRunsPolicy由调用线程执行任务（降级方案）
DiscardPolicy静默丢弃任务
DiscardOldestPolicy丢弃队列最老的任务，重新提交
⚠️ 线程池使用规范 阿里开发手册禁止使用 Executors 创建线程池。newFixedThreadPool 和 newSingleThreadExecutor 的 LinkedBlockingQueue 是无界的，可能导致 OOM；newCachedThreadPool 的 maximumPoolSize 是 Integer.MAX_VALUE，可能创建过多线程。

AQS（AbstractQueuedSynchronizer）原理

AQS 是 Java 并发包的基石，ReentrantLock、Semaphore、CountDownLatch 等都基于它实现。

  • 核心状态：一个 volatile int state 变量（0 表示未锁定，>0 表示锁定/重入次数）
  • CLH 队列：双向链表实现的 FIFO 等待队列
  • 独占模式：tryAcquire → addWaiter → acquireQueued
  • 共享模式：tryAcquireShared（用于 CountDownLatch、Semaphore 等）

volatile 关键字

volatile 保证两个特性：

  1. 可见性：修改后立即刷新到主内存，其他线程读取时从主内存加载
  2. 有序性：禁止指令重排序（通过内存屏障实现）

volatile 不保证原子性，例如 volatile int count; count++ 不是线程安全的，因为 count++ 实际是 read-modify-write 三步操作。

// DCL 双重检查锁单例（volatile 防止指令重排）
public class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                    // 不加 volatile，可能重排序为：
                    // 1. 分配内存 2. 引用赋值 3. 初始化
                    // 其他线程可能拿到未初始化的对象
                }
            }
        }
        return instance;
    }
}

ThreadLocal 原理与内存泄漏

每个 Thread 内部持有一个 ThreadLocalMap，key 是 ThreadLocal 对象（弱引用），value 是存储的值。

  • 内存泄漏原因：ThreadLocal 对象被 GC 后，key 变为 null，但 value 被 Entry 强引用无法回收
  • 解决方案：使用完毕后调用 threadLocal.remove()
  • 典型应用：数据库连接、用户上下文、日期格式化

垃圾回收算法

对象存活判断

  • 引用计数法：每个对象维护引用计数，有循环引用问题（JVM 不使用）
  • 可达性分析：从 GC Roots 出发，不可达的对象判定为可回收

GC Roots 包括：虚拟机栈中引用的对象、静态变量引用的对象、常量引用的对象、JNI 引用的对象等。

Virtual Thread 虚拟线程（Java 21 LTS）

虚拟线程是 JDK 21 正式发布的最重要特性。它是由 JVM 调度的轻量级线程，不绑定操作系统线程，可以轻松创建数百万个虚拟线程。

平台线程 vs 虚拟线程

对比项平台线程（Platform Thread）虚拟线程（Virtual Thread）
实现1:1 映射 OS 线程M:N 映射，由 ForkJoinPool 调度
内存开销每个线程 ~1MB 栈空间每个虚拟线程 ~几KB
创建数量通常数千个可创建数百万个
适用场景计算密集型IO 密集型（HTTP、数据库、文件）
上下文切换OS 内核态切换，开销大JVM 用户态切换，开销极小

创建方式

// 方式一：直接创建
Thread vt = Thread.ofVirtual().name("my-vt").start(() -> {
    System.out.println("Hello from virtual thread: " + Thread.currentThread());
});

// 方式二：虚拟线程 ExecutorService（推荐）
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            // 模拟 IO 操作，虚拟线程会自动 unmount 平台线程
            Thread.sleep(Duration.ofSeconds(1));
            System.out.println("Done: " + Thread.currentThread());
        });
    }
} // 自动关闭，等待所有任务完成

// 方式三：Spring Boot 3.x 配置启用虚拟线程
# application.yml
# spring.threads.virtual.enabled: true
⚠️ 面试高频：虚拟线程注意事项
  • 不要池化虚拟线程：每个任务创建一个虚拟线程，不要用线程池复用
  • 避免 synchronized 持有锁时做阻塞 IO：会导致 pinning（虚拟线程被固定到平台线程），改用 ReentrantLock
  • CPU 密集型任务不适合：虚拟线程的优势在于 IO 等待时释放平台线程
  • ThreadLocal 仍可使用但需谨慎：百万虚拟线程 + ThreadLocal = 巨大内存开销，建议用 Scoped Values 替代
// ❌ 错误：synchronized 内阻塞会导致 pinning
synchronized (lock) {
    httpClient.send(request, body); // 虚拟线程被固定，无法释放平台线程
}

// ✅ 正确：用 ReentrantLock 替代
reentrantLock.lock();
try {
    httpClient.send(request, body); // 虚拟线程可以 unmount
} finally {
    reentrantLock.unlock();
}

// ✅ JVM 参数检测 pinning（开发环境使用）
// -Djdk.tracePinnedThreads=full

Structured Concurrency（结构化并发，Preview）

结构化并发将多线程任务组织为有层级的任务树，父任务等待所有子任务完成后再继续。解决了 ExecutorService 缺乏结构化管理的问题。

// 传统方式：任务之间无结构关系，容易泄漏
var executor = Executors.newCachedThreadPool();
Future<User> userFuture = executor.submit(() -> findUser(id));
Future<Order> orderFuture = executor.submit(() -> fetchOrder(id));
// 如果 findUser 抛异常，fetchOrder 仍在后台运行（泄漏）

// 结构化并发：自动管理子任务生命周期
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<User> userTask = scope.fork(() -> findUser(id));
    Subtask<Order> orderTask = scope.fork(() -> fetchOrder(id));

    scope.join();            // 等待所有子任务完成
    scope.throwIfFailed();   // 任一失败则抛出异常，自动取消其他任务

    return new UserOrder(userTask.get(), orderTask.get());
}
💡 ShutdownOnFailure vs ShutdownOnSuccess
  • ShutdownOnFailure：任一子任务失败时取消所有其他任务（适合需要全部成功的场景）
  • ShutdownOnSuccess：任一子任务成功时取消其余任务（适合竞速场景，如多数据源取最快）

Scoped Values（作用域值，Preview）

Scoped Values 是 ThreadLocal 的替代方案，专为虚拟线程设计。它不可变、自动清理、无内存泄漏风险。

// 定义 Scoped Value
private static final ScopedValue<String> CURRENT_USER = ScopedValue.newInstance();

// 在作用域内绑定值
ScopedValue.runWhere(CURRENT_USER, "张三", () -> {
    handleRequest(); // 整个调用链都可以读取 CURRENT_USER
});

// 读取值
void handleRequest() {
    String user = CURRENT_USER.get(); // "张三"
    // 如果在作用域外调用 get()，抛出 NoSuchElementException
}

Sequenced Collections（有序集合，Java 21）

JDK 21 引入了 SequencedCollection、SequencedSet、SequencedMap 接口，统一了有序集合的操作方式。

// 新增的统一接口方法
interface SequencedCollection<E> extends Collection<E> {
    void addFirst(E e);       // 头部插入
    void addLast(E e);        // 尾部插入
    E getFirst();              // 获取第一个元素
    E getLast();               // 获取最后一个元素
    E removeFirst();           // 移除第一个
    E removeLast();            // 移除最后一个
    SequencedCollection<E> reversed(); // 返回反转视图
}

// 使用示例
List<String> list = new ArrayList<>(List.of("A", "B", "C"));
list.addFirst("Z");        // [Z, A, B, C]
list.addLast("D");         // [Z, A, B, C, D]
String first = list.getFirst(); // "Z"

// TreeMap 也支持
TreeMap<Integer, String> map = new TreeMap<>();
map.put(1, "一"); map.put(2, "二"); map.put(3, "三");
Map.Entry<Integer, String> first = map.firstEntry(); // 1=一
Map.Entry<Integer, String> last = map.lastEntry();   // 3=三
SequencedMap<Integer, String> reversed = map.reversed(); // 反转视图

Pattern Matching for switch（Preview）

扩展 switch 表达式支持模式匹配，消除冗长的 if-else instanceof 链。

// 传统方式：冗长的 instanceof 链
if (obj instanceof Integer i) {
    System.out.println("整数: " + i);
} else if (obj instanceof String s && s.length() > 0) {
    System.out.println("非空字符串: " + s);
} else if (obj instanceof List<?> list && list.size() > 3) {
    System.out.println("大列表: " + list.size());
}

// Pattern Matching for switch（JDK 21 Preview）
switch (obj) {
    case Integer i                    -> System.out.println("整数: " + i);
    case String s when s.length() > 0 -> System.out.println("非空字符串: " + s);
    case List<?> l when l.size() > 3 -> System.out.println("大列表: " + l.size());
    case null                         -> System.out.println("null");
    default                           -> System.out.println("其他");
}

// 配合 Sealed Class 实现穷举检查
sealed interface Shape permits Circle, Rectangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double w, double h) implements Shape {}

// 编译器知道 Shape 只有 Circle 和 Rectangle，无需 default
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius() * c.radius();
    case Rectangle r -> r.w() * r.h();
};

📡 IO 与 NIO

BIO vs NIO vs AIO

模型特点适用场景
BIO（同步阻塞）一个连接一个线程，read 阻塞连接数少且固定的场景
NIO（同步非阻塞）Channel + Buffer + Selector 多路复用高并发连接（聊天、推送）
AIO（异步非阻塞）回调通知，Proactor 模式大量长连接（文件服务器）

NIO 核心组件

  • Channel：双向数据通道（FileChannel、SocketChannel、ServerSocketChannel）
  • Buffer：数据缓冲区（ByteBuffer 最常用），capacity/limit/position 三个核心属性
  • Selector：多路复用器，一个线程管理多个 Channel
💡 Netty Netty 是基于 NIO 的高性能网络框架，面试中常问：线程模型（Reactor 模式）、零拷贝（CompositeByteBuf、FileRegion）、内存管理（池化 ByteBuf）。

🔮 反射与注解

反射原理

Java 反射在运行时获取类的信息并操作类的属性和方法。核心类在 java.lang.reflect 包下：

  • Class<?>：类的元信息
  • Field：成员变量
  • Method：方法
  • Constructor：构造器
// 获取 Class 对象的三种方式
Class<?> c1 = Class.forName("com.example.User");
Class<?> c2 = User.class;
Class<?> c3 = new User().getClass();

// 反射创建实例并调用方法
Object obj = c1.getDeclaredConstructor().newInstance();
Method method = c1.getDeclaredMethod("setName", String.class);
method.setAccessible(true);  // 突破 private 限制
method.invoke(obj, "张三");

注解原理

注解本质是一个继承了 java.lang.annotation.Annotation 的接口。运行时注解通过动态代理实现，编译时注解通过 APT（Annotation Processing Tool）在编译期生成代码。

Spring AOP 底层大量使用了反射和注解，理解反射机制有助于理解框架原理。
